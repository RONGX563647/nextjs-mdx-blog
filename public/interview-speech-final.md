## 第四页：Hutool 开源贡献（1 分钟）

### 技术栈
> Apache POI、OFD（ofdrw）、正则表达式、渲染器模式

### 痛点
> 接到文档生成需求，发现 Hutool 没有 Word 模板功能。现有方案要么功能单一，要么依赖太重。

### 技术方案

**Word 模板引擎**：
> 核心原理：`.docx` 本质是 ZIP 压缩包，包含 `word/document.xml`。
> 
> 实现思路：
> 1. POI 解压 ZIP，解析 XML DOM 树
> 2. 正则匹配 `{{name}}` / `{{@logo}}` / `{{#table}}` 三种占位符
> 3. 文本转义后替换（`<` → `&lt;` 防 XML 注入）
> 4. 重新打包

**技术亮点**：
> - 渲染器模式：TextRenderer / ImageRenderer / TableRenderer 三种策略
> - 链式 API 设计，开发者友好

### Bug 与解决

**Bug 1**：替换文本时特殊字符破坏 XML 结构
> **解决**：XML 转义，`<` → `&lt;`，`&` → `&amp;`

**Bug 2**：PR 被 maintainer 打回 3 次，说 API 设计不优雅
> **解决**：改为链式调用，最终合并

**PDF 生成**：
> 走捷径：基于 OFD 中间格式转换，复用 ofdrw 库，零额外依赖。

## 第五页：LightSSM（手写 Spring）（3 分钟）

### 技术栈
> Java 反射、JDK/CGLIB 动态代理、ASM、SpringMVC 前端控制器模式

### 动机
> 学 Spring 时困惑：注解、配置到底怎么工作的？几百万行源码无从下手，决定手写简化版。

### 核心一：IOC 容器（三级缓存）

**问题本质**：
> A 依赖 B，B 依赖 A，怎么创建？

**三级缓存设计**：
> - 一级：singletonObjects（完整 Bean）
> - 二级：earlySingletonObjects（早期引用，未初始化）
> - 三级：singletonFactories（ObjectFactory，延迟创建代理）
>
> **为什么三级？**
> AOP 代理需要在 Bean 初始化前创建。如果只有两级，循环依赖时无法暴露代理对象。三级通过 ObjectFactory 延迟创建代理。
>
> **执行流程**：
> 1. 创建 A，发现依赖 B → 暂停 A
> 2. 创建 B，发现依赖 A → 查三级缓存
> 3. 获取 A 的 ObjectFactory，调用 `getObject()` 提前暴露引用
> 4. B 拿到 A 的引用，完成创建
> 5. A 再拿到 B，完成创建
>
> **我的不足**：
> - 未实现 BeanPostProcessor 完整生命周期
> - 循环依赖仅限 setter 注入

### 核心二：SpringMVC

**DispatcherServlet 流程**：
> 请求 → DispatcherServlet → HandlerMapping → HandlerAdapter → Controller → 渲染
>
> **关键设计**：
> - HandlerMethodArgumentResolver 策略模式，支持 100+ 参数类型
> - HttpMessageConverter 负责对象转 JSON

### 核心三：AOP（拦截器链）

**执行流程**：
> 代理对象 invoke() → 获取拦截器链 → @Before → @Around → 目标方法 → @After
>
> **核心设计**：
> 所有拦截器共用 MethodInvocation，通过 `proceed()` 串联成责任链。

### Bug 与解决

**Bug 1**：三级缓存只用两级，循环依赖时 AOP 代理创建失败
> **解决**：引入第三级 ObjectFactory，延迟创建代理

**Bug 2**：JDK 代理只能代理接口，无接口类报错
> **解决**：实现 JDK/CGLIB 自动切换逻辑

**Bug 3**：MyBatis 整合时，事务不生效
> **解决**：实现 DataSourceTransactionManager 代理，拦截 SQL 执行

## 第六页：智能插座系统（核心重点）（8 分钟）⭐⭐⭐

### 技术栈
> **后端**：Java 21、Spring Boot 3.2、JPA、Redis、RabbitMQ、MQTT
> **前端**：Vue 3、UniApp
> **运维**：Prometheus、Grafana、Nginx
> **安全**：JWT、PBKDF2、RBAC

### 业务背景
> 宿舍电力设备缺乏监控，设备故障发现滞后。做 IoT 监控平台，支持千级设备并发。

### 痛点一：千级设备并发 OOM（1.5 分钟）

**问题本质**：
> 传统线程模型：1000 设备 = 1000 线程 = 1GB 内存（栈大小 1MB）

**方案对比**：
> 1. **Netty NIO**：性能好，但编程模型复杂，调试困难
> 2. **虚拟线程**（选型）：同步写代码，异步性能
>
> **虚拟线程原理**：
> - 平台线程：操作系统内核线程，栈固定 1MB
> - 虚拟线程：JVM 调度，栈动态扩展（初始几百字节）
> - 虚拟线程阻塞时，JVM 自动切换到其他虚拟线程
> - 适用 IO 密集型（网络/DB），不适用 CPU 密集型
>
> **效果**：内存 1GB → 几 MB，并发 1000 → 10,000+

### 痛点二：批量上报阻塞（2 分钟）

**数据量计算**：
> 1000 设备 × 5 秒/次 = 200 条/秒，同步写数据库连接池必爆。

**异步架构**：
> ```
> 设备 → MQTT → RabbitMQ → 异步消费 → 批量落库
> ```
>
> **消息可靠性保障**：
> 1. 生产者 Confirm：Broker 确认后认为发送成功
> 2. 消费者手动 ACK：业务成功后确认
> 3. 死信队列（DLQ）：失败消息转入，人工干预
> 4. 批量消费：100 条/批 + JDBC Batch 批量插入
>
> **效果**：吞吐 200 → 2000 条/秒（10 倍），DB IO 减少 80%

### 痛点三：接口响应慢（1.5 分钟）

**多级缓存架构**：
> - L1 Caffeine（本地）：热点数据，<1ms，5 分钟过期
> - L2 Redis（分布式）：全量数据，<10ms，10 分钟过期
> - 查询流程：L1 → L2 → DB → 回写 L2 → 回写 L1
>
> **缓存一致性**：
> Cache Aside 模式，**先更新 DB，再删除缓存**（不是更新缓存！）
>
> **为什么先删不更新？**
> 三种方案对比：
> 1. 先更 DB，再更缓存：并发更新时，旧数据覆盖新数据
> 2. 先删缓存，再更 DB：删除后到更新前，有请求读到旧 DB 写缓存
> 3. 先更 DB，再删缓存（选型）：简单有效，允许短暂不一致，过期时间兜底
>
> **效果**：命中率 95%，响应 500ms → <50ms

### 痛点四：恶意请求（1.5 分钟）

**限流方案对比**：
> 1. 单机（Guava）：多实例无法共享配额
> 2. 分布式（Redis+Lua）：多实例共享，有网络开销
> 3. 双层（Guava + Redis）（选型）：单机兜底 + 分布式精确
>
> **滑动窗口原理**：
> Lua 脚本在 Redis 单线程执行，ZSET 存储请求时间戳，自动过期。
>
> **限流算法对比**：
> - 令牌桶：允许突发，适用保护下游
> - 漏桶：恒定速率，适用限制速率
> - 滑动窗口：精确限流，无临界问题（选型）
>
> **效果**：QPS 100 → 10,000+（100 倍）

### 其他核心技术（1 分钟）

**JWT + Redis 黑名单**：
> JWT 无状态无法失效，Redis 存黑名单，过期时间 = JWT 剩余有效期（节省内存）。

**RBAC 权限**：
> 三级：接口级（API）+ 方法级（业务方法）+ 数据级（部门数据）。
> 数据权限动态拼接 SQL：AOP 拦截查询，自动拼接 `WHERE dept_id = 1001`。

**PBKDF2 加盐哈希**：
> - 随机盐值（每个用户独立）
> - 65536 次迭代（暴力破解成本增加 65536 倍）
> - 恒定时间比对（防时序攻击）

**Prometheus 监控**：
> 11 类指标（JVM/HTTP/DB/Redis/MQTT），8 个 Grafana 面板，秒级告警。

### Bug 与解决（1.5 分钟）

**Bug 1**：虚拟线程使用不当，创建太多导致调度器压力大
> **解决**：使用 `Executors.newVirtualThreadPerTaskExecutor()` 让 JVM 自动管理

**Bug 2**：RabbitMQ 消费者 ACK 前宕机，消息重新入队导致重复消费
> **解决**：消息唯一 ID + Redis 去重，实现幂等性

**Bug 3**：Redis 扣减成功，DB 订单创建失败 → 库存少了订单没生成
> **解决**：Redis 扣减后发延迟消息到 MQ，5 分钟后检查订单，没创建则回滚库存

**Bug 4**：Caffeine 缓存和 Redis 缓存数据不一致
> **解决**：L1 缓存设置较短过期时间（5 分钟），允许短暂不一致

## 第七页：不二价校园电商（4 分钟）

### 技术栈
> **后端**：Spring Cloud Alibaba（Nacos/Sentinel/Seata）、Redis、RabbitMQ
> **AI**：Spring AI、pgvector（向量数据库）
> **安全**：校园身份认证、地理围栏

### 痛点一：秒杀超卖（1.5 分钟）

**方案对比**：
> 1. 数据库乐观锁：`UPDATE stock SET count=count-1 WHERE id=1 AND count>0`
>    - 高并发下 CAS 失败率 30%（1000 QPS）
> 2. Redis Lua（选型）：无锁设计，失败率 0%（10000+ QPS）
>
> **效果**：库存超卖率 0%

### 痛点二：分布式事务（1 分钟）

**Seata AT 原理**：
> 1. 一阶段：执行 SQL 前查原始数据，记录 undo log（前后镜像），提交本地事务
> 2. 二阶段提交：异步删除 undo log
> 3. 二阶段回滚：根据 undo log 反向补偿
>
> **对比**：
> - TCC：性能高但侵入性强
> - Saga：支持长事务但不支持隔离性
> - AT（选型）：无侵入，开发效率高

### 痛点三：AI 智能客服（1 分钟）

**RAG 核心**：检索增强生成（Retrieval-Augmented Generation）
> 1. 知识库：FAQ 转 Embedding 向量，存入 pgvector
> 2. 检索：用户问题转向量，余弦相似度搜索 Top-K
> 3. 生成：知识作为上下文，大模型生成答案
>
> **为什么不用纯大模型？**
> 大模型会"幻觉"，知识库保证准确性，更新知识库比微调成本低。
>
> **优化**：多轮对话上下文（最近 5 轮），AI 能理解上下文。
>
> **效果**：AI 客服覆盖率 60%

### 痛点四：校外用户混入（30 秒）

**解决方案**：
> - 学号 + 密码认证（对接学校统一身份认证）
> - 地理围栏：仅限校内 IP 段注册
> - 效果：100% 校内用户

### Bug 与解决

**Bug 1**：Seata AT 模式下，一阶段持锁时间过长，影响并发
> **解决**：优化 SQL，减少查询时间，快速提交本地事务

**Bug 2**：Sentinel 限流规则配置不当，正常请求被误杀
> **解决**：调整 QPS 阈值，设置预热时间（Warm Up）

**Bug 3**：AI 客服返回答案不准确
> **解决**：优化 Embedding 模型，调整相似度阈值，增加知识库覆盖面
