### #2 重点

**集合框架**：`Collection`（单列）与`Map`（双列）为核心。`ArrayList`（数组结构，查询快）、`LinkedList`（链表，增删快）；`HashSet`（哈希表，元素唯一）、`LinkedHashSet`（有序）；`HashMap`（线程不安全，支持 null）、`ConcurrentHashMap`（并发场景）。迭代器`Iterator`遍历集合，增强`for`简化操作。

**Stream 流**：链式操作处理集合，`filter`过滤、`map`转换、`collect`收集，支持并行计算（`parallelStream()`）。

**设计模式**：工厂模式解耦对象创建，单例模式确保全局唯一（枚举 / 静态内部类实现），代理模式控制访问，观察者模式实现订阅通知。

**其他**：不可变集合（`List.of`）防篡改，`File`操作路径，NIO.2`Paths`/`Files`处理文件读写。

### 33.File类

> `java.io.File` 类用于操作文件或目录的路径，不涉及内容读写，核心功能如下：
>
> #### **1. 构造方法**
>
> - **单路径**：`new File("路径字符串")`，如 `new File("d:/a.txt")`。
> - **父子路径**：`new File("父路径", "子路径")` 或结合 `File` 对象创建，支持相对路径（相对于项目根目录）和绝对路径。
>
> #### **2. 核心功能**
>
> - 获取信息
>
>   ：
>
>   - `getAbsolutePath()`：获取绝对路径（如 `D:\test\a.txt`）。
>   - `getName()`：获取名称（如 `a.txt`）。
>   - `length()`：获取文件大小（目录长度未定义）。
>
> - 判断功能
>
>   ：
>
>   - `exists()`：判断是否存在。
>   - `isFile()`/`isDirectory()`：判断是文件或目录。
>
> - 创建 / 删除
>
>   ：
>
>   - `createNewFile()`：创建新文件（需父目录存在）。
>   - `mkdir()`/`mkdirs()`：创建单级 / 多级目录。
>   - `delete()`：删除文件或空目录（非空目录需递归删除）。
>
> #### **3. 目录遍历**
>
> - `list()`：返回子文件 / 目录名（`String[]`）。
> - `listFiles()`：返回子文件 / 目录对象（`File[]`），遍历需判断 `isFile()`/`isDirectory()`。
> - **递归遍历**：处理多级目录时，对每个子目录递归调用遍历方法。
>
> #### **4. 典型场景**
>
> - **文件操作**：判断文件存在性、获取文件属性。
> - **目录管理**：创建多级目录、删除非空目录（需先删子内容）。
> - **综合应用**：统计文件夹总大小（累加所有文件大小）、按后缀名统计文件类型（结合 `HashMap`）。
>
> #### **注意事项**
>
> - 路径分隔符用 `\\`（Windows）或 `/`（通用），推荐用 `File.separator` 保证跨平台。
> - 操作前需用 `exists()` 判断是否存在，避免空指针异常。
> - 目录遍历需处理 `listFiles()` 返回 `null` 的情况（路径不存在或无权限）。



### 34.递归思想

> ### **递归思想的本质**
>
> 递归的核心是**将复杂问题分解为同构的简单问题**，利用数学归纳法的思维：
>
> - **基础情况**：解决最小规模的问题（如 `n=0`）。
> - **归纳假设**：假设能解决规模为 `n-1` 的问题。
> - **归纳步骤**：用假设解决规模为 `n` 的问题。
>
> 通过这种 “自顶向下” 的分解，递归能以简洁的代码解决看似复杂的问题，但需注意性能和栈溢出风险。在实际开发中，可先用递归理清逻辑，再根据需求优化为迭代实现。

递归是一种 “**以简驭繁**” 的思维工具，其本质是通过数学归纳法将复杂问题转化为可解的简单问题。掌握递归需：

1. 明确终止条件与递推关系（避免逻辑错误）。

2. 理解调用栈原理（避免栈溢出）。

3. 权衡性能与代码简洁性（选择递归或迭代）。
   从算法设计到系统架构（如微服务链式调用），递归思想贯穿计算机科学，其核心是 “**分解 - 解决 - 合并**” 的问题求解范式，体现了人类对复杂系统的抽象控制能力。

   

4. | **维度**       | **递归**                        | **迭代**                                                 |
   | -------------- | ------------------------------- | -------------------------------------------------------- |
   | **抽象层次**   | 高层抽象（关注 “做什么”）       | 低层实现（关注 “如何做”）                                |
   | **代码复杂度** | 简洁（少循环变量）              | 可能冗长（需维护索引、终止条件）                         |
   | **空间效率**   | 可能低效（栈帧开销）            | 通常高效（可复用变量，如斐波那契迭代仅需 \(O(1)\) 空间） |
   | **尾递归优化** | 理论可优化为迭代（Java 不支持） | 天然迭代实现                                             |
   | **调试难度**   | 难（多层栈帧跟踪）              | 易（单线程顺序执行）                                     |

### 36.字节流

> **字节流（Byte Stream）**
>
> - 基类：`InputStream` 和 `OutputStream`
> - 适用场景：处理二进制文件（如图片、视频）、网络通信等
> - 关键子类：
>   - `FileInputStream`/`FileOutputStream`：文件读写
>   - `BufferedInputStream`/`BufferedOutputStream`：带缓冲的字节流
>   - `DataInputStream`/`DataOutputStream`：处理基本数据类型

- **InputStream**：
  - `int read()`：读取一个字节（返回 0-255 的整数，EOF 返回 - 1）。
  - `int read(byte[] b)`：读取最多`b.length`个字节到数组，返回实际读取的字节数。
  - `void close()`：关闭流释放资源（需手动调用或用`try-with-resources`）。
- **OutputStream**：
  - `void write(int b)`：写入一个字节（参数的低 8 位有效）。
  - `void write(byte[] b)`：写入整个字节数组。
  - `void flush()`：强制刷新缓冲区（确保数据写入目标）。
  - `void close()`：关闭流（自动调用`flush()`）。

| **特性** | **字节流（InputStream/OutputStream）**    | **字符流（Reader/Writer）**    |
| -------- | ----------------------------------------- | ------------------------------ |
| 处理单位 | 字节（8 位）                              | 字符（16 位 Unicode）          |
| 适用场景 | 二进制文件（图片、音频）                  | 文本文件（.txt、.java）        |
| 编码处理 | 需手动处理编码                            | 自动处理编码（UTF-8 等）       |
| 典型子类 | `FileInputStream`, `BufferedOutputStream` | `FileReader`, `BufferedWriter` |

```markdown

Java字节流是处理二进制数据的核心IO机制，基于`InputStream`和`OutputStream`抽象类，以8位字节为单位读写数据，适用于图片、音频等任意文件类型。  

**核心类**：  
- **输入流**：`FileInputStream`（文件输入）、`BufferedInputStream`（带缓冲，提升性能）、`DataInputStream`（读取基本类型）。  
- **输出流**：`FileOutputStream`（文件输出）、`BufferedOutputStream`（缓冲输出）、`DataOutputStream`（写入基本类型）。  

**关键操作**：  
- **读写方法**：`read()`/`write(int b)`逐字节操作，`read(byte[])`/`write(byte[])`批量操作（推荐用缓冲数组提升效率）。

- **追加写入**：创建`FileOutputStream`时传入`true`参数（如`new FileOutputStream("a.txt", true)`）。

- **文件复制**：通过缓冲数组循环读写（如`byte[] buffer = new byte[1024]`），减少IO次数。  
- **类型读写**：`DataInputStream`/`DataOutputStream`按固定格式读写基本类型（需注意顺序一致）。  

**与字符流区别**：字节流直接处理二进制数据，字符流需结合编码处理文本（如UTF-8），后者本质是字节流的高层封装。  

总结：字节流是IO基础，掌握缓冲流和装饰器模式可灵活处理各类二进制数据，注意关闭资源和异常处理。
```

### 37.字符流

```markdown

Java字符流以**16位Unicode字符**为单位处理文本数据，基于`Reader`和`Writer`抽象类，自动处理字符编码（如UTF-8/GBK），适用于纯文本文件（.txt/.java等）。  

**核心类**：  
- **输入流**：`FileReader`（文件字符输入）、`BufferedReader`（带缓冲，支持按行读取）。  
- **输出流**：`FileWriter`（文件字符输出）、`BufferedWriter`（缓冲输出，支持换行）。

- **读写方法**：`read()`逐字符读，`read(char[])`批量读；`write(String)`写字符串，`write(char[])`写字符数组。  
- **编码处理**：构造流时可指定编码（如`new FileReader("a.txt", "UTF-8")`），避免乱码。  
- **资源管理**：需调用`close()`释放资源，`flush()`刷新缓冲区确保数据写入。  

**典型场景**：  
- **文本读写**：直接操作字符，如`FileWriter.write("Hello")`，无需手动处理字节与字符转换。  
- **按行读写**：搭配`BufferedReader.readLine()`高效处理大文本（如日志文件）。  

**与字节流区别**：  
- 字节流处理**所有二进制数据**（图片/音频等），字符流仅处理**文本数据**；  
- 字符流本质是字节流+编码转换，简化了文本处理流程。  

总结：处理纯文本时优先用字符流（便捷性+编码自动处理），其他场景（如图像/二进制）必须用字节流。注意关闭流和编码一致性。
```

### 43.装饰器 ？

Java 装饰器（Decorator）是一种结构型设计模式，核心是通过包装对象动态扩展功能，而不修改原有类代码。
**特点**：

- 装饰类与被装饰类实现相同接口，保证透明性。

- 可多层嵌套装饰，灵活组合功能（如 IO 流中`BufferedReader`包装`InputStreamReader`）。
  **优势**：避免继承导致的类爆炸，符合开闭原则，便于功能扩展与复用。

  ~~~markdown
  ### Java装饰器模式：核心原理与实战解析
  
  Java装饰器模式是一种**结构型设计模式**，允许在**不修改原有对象**的前提下，动态地给对象添加额外职责。它通过组合而非继承来扩展功能，使代码更具灵活性和可维护性。
  
  
  #### **核心原理**
  1. **抽象组件（Component）**  
     定义被装饰对象的接口，装饰器和具体组件必须实现该接口。  
     ```java
     public interface Beverage {
         String getDescription();
         double cost();
     }
     ```
  
  2. **具体组件（Concrete Component）**  
     实现抽象组件接口的基础对象，提供核心功能。  
     ```java
     public class Espresso implements Beverage {
         @Override
         public String getDescription() { return "Espresso"; }
         @Override
         public double cost() { return 1.99; }
     }
     ```
  
  3. **抽象装饰器（Decorator）**  
     继承抽象组件，并持有一个被装饰对象的引用，通过构造函数注入。  
     ```java
     public abstract class CondimentDecorator implements Beverage {
         protected Beverage beverage; // 持有被装饰对象
         public CondimentDecorator(Beverage beverage) {
             this.beverage = beverage;
         }
     }
     ```
  
  4. **具体装饰器（Concrete Decorator）**  
     扩展抽象装饰器，添加新功能或修改行为。  
     ```java
     public class Mocha extends CondimentDecorator {
         public Mocha(Beverage beverage) {
             super(beverage);
         }
         
         @Override
         public String getDescription() {
             return beverage.getDescription() + ", Mocha";
         }
         
         @Override
         public double cost() {
             return beverage.cost() + 0.20; // 附加费用
         }
     }
     ```
  
  
  #### **实战应用：Java IO流**
  Java IO流是装饰器模式的经典应用，通过多层装饰实现功能组合：
  
  ```java
  // 基础组件：FileInputStream（具体组件）
  InputStream fileInput = new FileInputStream("data.txt");
  
  // 第一层装饰：BufferedInputStream（添加缓冲功能）
  InputStream bufferedInput = new BufferedInputStream(fileInput);
  
  // 第二层装饰：DataInputStream（添加读取基本数据类型的功能）
  DataInputStream dataInput = new DataInputStream(bufferedInput);
  
  // 使用组合后的功能
  int num = dataInput.readInt(); // 读取int类型数据
  ```
  
  
  #### **关键特性**
  1. **动态组合**  
     运行时可动态选择装饰器组合，无需编译时确定。  
     ```java
     Beverage drink = new Espresso(); // 基础咖啡
     drink = new Mocha(drink);        // 添加摩卡
     drink = new Whip(drink);         // 再添加奶泡
     System.out.println(drink.cost()); // 动态计算总价
     ```
  
  2. **透明接口**  
     装饰器与被装饰对象实现相同接口，客户端无需关心具体实现。  
     ```java
     // 客户端统一使用Beverage接口调用
     void printBeverageInfo(Beverage beverage) {
         System.out.println(beverage.getDescription());
     }
     ```
  
  3. **开闭原则**  
     新增功能只需创建新的装饰器类，无需修改现有代码。
  
  
  #### **与继承的对比**
  | **维度**       | **装饰器模式**               | **继承**                     |
  |----------------|-----------------------------|-----------------------------|
  | **扩展方式**   | 组合（运行时动态扩展）       | 继承（编译时静态扩展）       |
  | **类数量**     | 更少（通过组合复用装饰器）   | 更多（每个功能组合需创建子类） |
  | **耦合度**     | 低（装饰器与组件松耦合）     | 高（子类依赖父类实现）       |
  | **灵活性**     | 高（可自由组合装饰器）       | 低（功能组合固定）           |
  
  
  #### **优缺点**
  - **优点**：  
    - 避免类爆炸，简化复杂继承结构。  
    - 支持动态功能扩展，适应需求变化。  
    - 符合开闭原则，代码更易维护。  
  
  - **缺点**：  
    - 多层装饰可能导致代码复杂度上升。  
    - 过度使用会使调用链变长，调试难度增加。  
  
  
  #### **应用场景**
  - **IO流**：Java的`InputStream`/`OutputStream`体系。  
  - **Servlet Filter**：Servlet规范中的过滤器链。  
  - **日志与事务**：为服务添加日志记录、事务管理等功能。  
  
  
  ### **总结**
  装饰器模式通过**组合替代继承**，在不修改原有对象的前提下实现功能扩展，使代码更具弹性。它的核心在于**透明包装**和**动态组合**，适用于需要灵活扩展功能、避免继承滥用的场景。掌握装饰器模式是理解Java IO、Servlet等框架的基础。
  ~~~

Java 编码解码的核心是**字符集与编码的匹配**。通过`String.getBytes()`、`new String()`、转换流等工具，开发者可实现字符与字节的双向转换。关键是全程保持编码一致（推荐 UTF-8），并在处理外部数据（如文件、网络）时显式指定编码，避免依赖系统默认设置。掌握这些机制可有效解决乱码问题，确保数据正确传输与存储。
