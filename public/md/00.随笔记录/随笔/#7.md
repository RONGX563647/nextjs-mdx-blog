##### 示例 1：ArrayList vs LinkedList（八股文必问）

java

```java
// ArrayList底层是动态数组（数组+扩容机制）
List<String> arrayList = new ArrayList<>();
// LinkedList底层是双向链表（节点+指针）
List<String> linkedList = new LinkedList<>();

/*
  面试考点：
  1. 增删效率：
     - ArrayList：尾部增删O(1)，中间增删需移动元素（O(n)）
     - LinkedList：中间增删只需修改指针（O(1)），但查询需遍历（O(n)）
  2. 扩容机制：
     - ArrayList初始容量10，扩容为1.5倍（int newCapacity = oldCapacity + (oldCapacity >> 1)）
     - LinkedList无固定容量，无需扩容
  3. 内存占用：
     - ArrayList连续内存，可能有冗余空间（扩容预留）
     - LinkedList每个节点存前后指针，内存开销更大
  4. 线程安全：
     - 两者均线程不安全，多线程场景需用Collections.synchronizedList()或CopyOnWriteArrayList
*/
```

##### 示例 2：HashMap 底层原理（JDK8+）

java











```java
Map<String, Integer> hashMap = new HashMap<>();
hashMap.put("key", 1);

/*
  面试考点：
  1. 数据结构：数组+链表+红黑树（JDK8引入红黑树优化长链表）
     - 当链表长度>8且数组容量>=64时，链表转为红黑树（查询效率从O(n)→O(log n)）
     - 红黑树节点数<6时，退化为链表（红黑树维护成本高）
  2. 哈希冲突解决：链地址法（数组索引相同的元素用链表/红黑树串联）
  3. 扩容机制：
     - 初始容量16，负载因子0.75，阈值=16*0.75=12（元素数>阈值时扩容）
     - 扩容为2倍（保证容量是2^n，支持位运算计算索引：index = (n-1) & hash）
  4. 线程安全：线程不安全（多线程put可能导致链表成环），需用ConcurrentHashMap
*/
```

##### 示例：IO 流的装饰器模式（面试常考设计思想）

// 节点流：直接操作数据源（文件）
FileInputStream fis = new FileInputStream("test.txt");

#### 3. 多线程（核心中的核心）

**核心考点**：线程创建方式、线程状态、同步机制（synchronized/volatile/Lock）、线程池参数。

##### 示例 1：synchronized 与 Lock 的区别（八股文高频）

java











```java
// 1. synchronized（隐式锁，JVM层面实现）
synchronized void syncMethod() {
    // 同步代码块
}

// 2. ReentrantLock（显式锁，API层面实现）
Lock lock = new ReentrantLock();
void lockMethod() {
    lock.lock();
    try {
        // 同步代码块
    } finally {
        lock.unlock(); // 必须手动释放
    }
}

/*
  面试考点：
  1. 锁释放：synchronized自动释放（异常/方法结束），Lock需手动释放（finally中）
  2. 锁类型：synchronized是非公平锁，ReentrantLock可指定公平/非公平（构造参数）
  3. 功能：Lock支持中断响应（lockInterruptibly()）、超时获取（tryLock()）、条件变量（newCondition()）
  4. 性能：JDK6后synchronized优化（偏向锁→轻量级锁→重量级锁），与Lock性能接近
*/
```

##### 示例 2：线程池参数及核心原理（场景题基础）

java











```java
// 企业开发常用线程池（不推荐Executors，手动创建更可控）
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2, // 核心线程数（常驻线程，即使空闲也不销毁）
    5, // 最大线程数（核心线程+临时线程）
    60L, // 临时线程空闲时间
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(10), // 任务队列（核心线程满后存这里）
    Executors.defaultThreadFactory(),
    new ThreadPoolExecutor.AbortPolicy() // 拒绝策略（队列满+线程达最大时触发）
);

/*
  面试考点：
  1. 任务执行流程：
     核心线程未满→直接执行 → 核心线程满→入队列 → 队列满→创建临时线程 → 临时线程达最大→触发拒绝策略
  2. 拒绝策略：
     - AbortPolicy：抛异常（默认）
     - CallerRunsPolicy：让提交任务的线程自己执行（缓解压力）
     - DiscardOldestPolicy：丢弃队列中最老的任务
  3. 企业场景：根据任务类型（CPU密集/IO密集）调整参数
     - CPU密集：核心线程数=CPU核心数+1（避免频繁上下文切换）
     - IO密集：核心线程数=CPU核心数*2（线程多数时间在等IO，可多开）
*/
```

### 三、场景题示例（结合实际开发）

场景题是理论的 “落地应用”，考察你能否用基础知识点解决实际问题。

#### 场景 1：如何高效读取 1GB 大文件（IO 流场景题）

**问题**：直接用 FileInputStream 读取大文件可能内存溢出，如何优化？

**解决方案**：

java











```java
// 方案1：Java原生（缓冲流+分段读取）
try (BufferedInputStream bis = new BufferedInputStream(new FileInputStream("bigFile.txt"));
     BufferedOutputStream bos = new BufferedOutputStream(new FileOutputStream("copy.txt"))) {
    
    byte[] buffer = new byte[8192]; // 8KB缓冲区（避免一次读1GB）
    int len;
    while ((len = bis.read(buffer)) != -1) { // 分段读取
        bos.write(buffer, 0, len); // 分段写入
    }
} catch (IOException e) {
    e.printStackTrace();
}

// 方案2：Hutool工具类（一行搞定，底层也是分段读取）
FileUtil.copy("bigFile.txt", "copy.txt", true);

/*
  场景题考察点：
  1. 缓冲流的作用：减少IO次数（内存与磁盘交互昂贵，一次读8KB比一次读1字节高效）
  2. 资源释放：try-with-resources自动关闭流，避免内存泄漏
  3. 工具类使用：企业开发不会重复造轮子，Hutool/Commons IO已封装最佳实践
*/
```

#### 场景 2：多线程并发修改 ArrayList 导致异常（集合场景题）

**问题**：多个线程同时向 ArrayList 添加元素，会出现`ConcurrentModificationException`，如何解决？

**解决方案**：

java

```java
// 方案1：使用线程安全的集合（CopyOnWriteArrayList）
List<String> safeList = new CopyOnWriteArrayList<>();

// 方案2：手动加锁（synchronized或Lock）
List<String> arrayList = new ArrayList<>();
synchronized (arrayList) {
    arrayList.add("element");
}

/*
  场景题考察点：
  1. 异常原因：ArrayList的迭代器是快速失败（fail-fast），检测到并发修改时抛异常
  2. CopyOnWriteArrayList原理：写时复制（修改时复制新数组，读不加锁，适合读多写少场景）
  3. 取舍：CopyOnWriteArrayList内存开销大（复制数组），高并发写场景推荐ConcurrentLinkedQueue
*/
```

#### 场景 3：如何用多线程优化批量任务（多线程场景题）

**问题**：有 1000 个文件需要解析，单线程效率低，如何用多线程优化？

**解决方案**：

java

```java
// 1. 创建线程池（根据CPU核心数设置线程数）
int corePoolSize = Runtime.getRuntime().availableProcessors() + 1;
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    corePoolSize,
    corePoolSize * 2,
    60L,
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000)
);

// 2. 提交1000个解析任务
for (int i = 0; i < 1000; i++) {
    int fileIndex = i;
    executor.submit(() -> {
        parseFile("file_" + fileIndex + ".txt"); // 解析单个文件的方法
    });
}

// 3. 关闭线程池
executor.shutdown();
executor.awaitTermination(1, TimeUnit.HOURS); // 等待所有任务完成

/*
  场景题考察点：
  1. 线程数设置：CPU密集型任务（解析文件偏CPU）线程数=核心数+1，避免上下文切换过多
  2. 任务拆分：将大任务拆分为小任务，并行执行
  3. 线程池关闭：shutdown()（拒绝新任务，等待已提交任务）+ awaitTermination()（等待完成）
*/
```

### 四、高效学习建议（针对你的问题）

#### 1. 解决 “看重复视频困、效率低”

- **用 “输出倒逼输入”**：看完一个知识点（如 ArrayList 底层），立即写一段代码验证（比如测试扩容机制），再尝试给 “假想面试官” 讲解原理（能讲清楚才是真懂）。
- **用 “场景题巩固”**：比如学完 IO 流，马上做 “大文件复制” 场景题，对比原生 API 和 Hutool 的实现，记录差异。

#### 2. 避免 “好高骛远”：基础不牢时的学习顺序

- **先 “能用” 再 “懂原理”**：比如多线程，先熟练用线程池提交任务，再深入理解核心参数；先会用 Hutool 的`FileUtil`，再去看底层 IO 流实现。
- **用 “项目驱动” 串联知识点**：比如写一个 “文件批量解析工具”，会用到 IO 流（读文件）、集合（存解析结果）、多线程（并行解析）、异常处理（文件不存在 / 格式错误），一举多得。

#### 3. 高效回顾：对抗遗忘的方法

- “3-7-15 天” 复习法

  ：

  - 当天：学完后用思维导图整理核心点（如 synchronized 的 3 个特性：可重入、不可中断、非公平）。
  - 第 3 天：做 1-2 道相关场景题（如 “如何实现一个线程安全的单例模式”）。
  - 第 7 天：对比相似知识点（如 ArrayList vs Vector），写一篇 “对比笔记”。

- **“错题本” 记录**：把容易混淆的点（如 “wait () 和 sleep () 的区别”）、做错的场景题整理成笔记，标注错误原因。

### 总结

Java 学习的核心是 “理论→实践→场景” 的循环：

- 八股文是理论基础（如集合底层、多线程锁机制），决定你能否 “说清楚”；
- 场景题是实践应用（如并发修改集合、线程池优化），决定你能否 “做出来”；
- 工具包（Hutool/Apache）是效率工具，企业开发中用它们替代重复代码，但底层原理仍需掌握（否则出问题无法排查）。

按 “先能用，再懂原理，最后练场景” 的节奏推进，既能避免好高骛远，又能高效巩固知识。

### 一、IO 流进阶（企业开发高频）

#### 1. 八股文：NIO 与传统 IO 的核心区别（面试区分度高）

java



```java
// 传统IO（BIO）：面向流，阻塞式
try (InputStream is = new FileInputStream("file.txt");
     OutputStream os = new FileOutputStream("copy.txt")) {
    byte[] buffer = new byte[1024];
    int len;
    while ((len = is.read(buffer)) != -1) { // 读取阻塞，直到有数据
        os.write(buffer, 0, len);
    }
}

// NIO：面向缓冲区，非阻塞（Selector+Channel）
try (FileChannel inChannel = new FileInputStream("file.txt").getChannel();
     FileChannel outChannel = new FileOutputStream("copy.txt").getChannel()) {
    ByteBuffer buffer = ByteBuffer.allocate(1024);
    while (inChannel.read(buffer) != -1) { // 非阻塞，无数据时返回0
        buffer.flip(); // 切换为读模式
        outChannel.write(buffer);
        buffer.clear(); // 清空缓冲区，准备下次写入
    }
}

/*
  核心区别（八股文必答）：
  1. 数据处理方式：
     - IO：面向流（一次读一个字节，无法回退）
     - NIO：面向缓冲区（数据先读入缓冲区，可重复读写）
  2. 阻塞特性：
     - IO：读写阻塞（线程等待数据，效率低）
     - NIO：非阻塞（线程可处理其他任务，通过Selector监听通道事件）
  3. 适用场景：
     - IO：简单场景（小文件读写）、同步阻塞需求
     - NIO：高并发场景（如服务器通信）、大文件处理（通道可映射内存）
*/
```

#### 2. 场景题：用多线程 + NIO 拆分 10GB 大文件（企业级文件处理）

java

```java
// 需求：将10GB文件按1GB拆分，多线程并行处理，用Hutool简化操作
public class BigFileSplitter {
    private static final int BUFFER_SIZE = 1024 * 1024; // 1MB缓冲区
    private static final long CHUNK_SIZE = 1024 * 1024 * 1024; // 1GB/分片

    public static void main(String[] args) throws IOException {
        String sourceFile = "10GBfile.dat";
        String targetDir = "chunks/";
        FileUtil.mkdir(targetDir); // Hutool：创建目录

        // 获取文件总大小
        long fileSize = FileUtil.size(sourceFile);
        int chunkCount = (int) Math.ceil((double) fileSize / CHUNK_SIZE);

        // 多线程拆分（线程池控制并发）
        ExecutorService executor = new ThreadPoolExecutor(
            4, // 核心线程数（根据CPU核心数调整）
            4,
            0L, TimeUnit.MILLISECONDS,
            new LinkedBlockingQueue<>()
        );

        for (int i = 0; i < chunkCount; i++) {
            int chunkIndex = i;
            executor.submit(() -> {
                splitChunk(sourceFile, targetDir, chunkIndex, CHUNK_SIZE);
            });
        }
        executor.shutdown();
    }

    // 拆分单个分片（NIO通道+缓冲区）
    private static void splitChunk(String sourceFile, String targetDir, int chunkIndex, long chunkSize) {
        try (FileChannel inChannel = FileChannel.open(Paths.get(sourceFile), StandardOpenOption.READ);
             FileChannel outChannel = FileChannel.open(
                 Paths.get(targetDir + "chunk_" + chunkIndex + ".dat"),
                 StandardOpenOption.CREATE, StandardOpenOption.WRITE
             )) {

            long start = chunkIndex * chunkSize;
            inChannel.position(start); // 定位到分片起始位置
            ByteBuffer buffer = ByteBuffer.allocateDirect(BUFFER_SIZE); // 直接缓冲区（堆外内存，减少拷贝）

            long bytesRead;
            long remaining = chunkSize;
            while (remaining > 0 && (bytesRead = inChannel.read(buffer)) != -1) {
                buffer.flip();
                long writeLen = Math.min(remaining, bytesRead);
                outChannel.write(buffer, 0, (int) writeLen); // 写入分片
                remaining -= writeLen;
                buffer.clear();
            }
            System.out.println("分片" + chunkIndex + "完成");
        } catch (IOException e) {
            throw new RuntimeException("分片失败：" + e.getMessage(), e); // 异常链：保留原始异常信息
        }
    }
}
```

**场景题核心考点**：

1. **NIO 优势**：使用`FileChannel`和直接缓冲区（`DirectByteBuffer`）减少 JVM 堆内存占用，适合大文件。
2. **多线程协作**：线程池控制并发数（避免线程过多导致 OOM），按文件偏移量拆分任务（无重叠）。
3. **工具类简化**：Hutool 的`FileUtil`替代手动创建目录、计算文件大小等重复代码。
4. **异常处理**：使用异常链（`throw new RuntimeException(..., e)`）保留原始异常栈，便于排查问题。

#### 2. 场景题：高并发下的缓存容器设计（避免 ConcurrentModificationException）

java

```java
// 需求：设计一个缓存容器，支持高并发读写，且迭代时不抛并发修改异常
public class SafeCache<K, V> {
    // 选择CopyOnWriteArrayList：读不加锁，写时复制新数组（适合读多写少场景）
    private final CopyOnWriteArrayList<CacheEntry<K, V>> cache = new CopyOnWriteArrayList<>();

    // 缓存条目：封装键值对
    private static class CacheEntry<K, V> {
        K key;
        V value;
        long expireTime; // 过期时间（毫秒）

        CacheEntry(K key, V value, long expireTime) {
            this.key = key;
            this.value = value;
            this.expireTime = expireTime;
        }
    }

    // 写入缓存（加时复制，保证线程安全）
    public void put(K key, V value, long expireSeconds) {
        long expireTime = System.currentTimeMillis() + expireSeconds * 1000;
        // 先删除旧值（CopyOnWriteArrayList的removeIf是原子操作）
        cache.removeIf(entry -> key.equals(entry.key));
        // 添加新值
        cache.add(new CacheEntry<>(key, value, expireTime));
    }

    // 读取缓存（无锁，高效）
    public V get(K key) {
        long now = System.currentTimeMillis();
        // 迭代时即使被修改，也操作旧数组（无ConcurrentModificationException）
        for (CacheEntry<K, V> entry : cache) {
            if (key.equals(entry.key) && entry.expireTime > now) { // 校验过期
                return entry.value;
            }
        }
        return null;
    }

    // 定时清理过期缓存（单独线程，避免阻塞读写）
    public SafeCache() {
        ScheduledExecutorService scheduler = Executors.newSingleThreadScheduledExecutor();
        scheduler.scheduleAtFixedRate(() -> {
            long now = System.currentTimeMillis();
            cache.removeIf(entry -> entry.expireTime <= now); // 原子删除过期条目
        }, 1, 5, TimeUnit.MINUTES); // 每5分钟清理一次
    }
}
```

**场景题核心考点**：

1. **容器选择**：`CopyOnWriteArrayList`适合读多写少场景（如缓存），迭代时无并发异常（迭代的是旧数组快照）。
2. **线程安全设计**：写入操作（`removeIf`+`add`）是原子的，读取无锁（避免`synchronized`的性能开销）。
3. **过期策略**：单独线程定时清理（非读写时阻塞），平衡性能与内存占用。

    }

// 正例2：try-with-resources自动关闭资源（避免内存泄漏）
public String readFile(String path) {
    try (BufferedReader reader = new BufferedReader(new FileReader(path))) { // 资源自动关闭
        return reader.lines().collect(Collectors.joining("\n"));
    } catch (FileNotFoundException e) {
        // 业务化异常：将IO异常转为业务异常（前端无需知道"文件不存在"，只需知道"资源不存在"）

2. **资源关闭**：`try-with-resources`自动关闭实现`AutoCloseable`的资源（如流、连接），避免手动`finally`关闭的遗漏。
3. **异常传递原则**：不吞异常、不捕获`Throwable`、用异常链保留原始信息（便于排查）。

#### 2. 场景题：多线程统计大集合数据（分片 + ThreadLocal）

java

```java
// 需求：统计100万条订单中各状态的数量（如"已支付"、"待发货"），用多线程加速
public class OrderStatistic {
    // 订单状态枚举
    public enum OrderStatus { PAID, UNPAID, SHIPPED, DELIVERED }

    // 线程私有计数器（避免同步锁竞争）
    private static ThreadLocal<Map<OrderStatus, Integer>> threadLocal = ThreadLocal.withInitial(() -> {
        Map<OrderStatus, Integer> map = new EnumMap<>(OrderStatus.class);
        // 初始化计数器为0
        for (OrderStatus status : OrderStatus.values()) {
            map.put(status, 0);
        }
        return map;
    });

    public Map<OrderStatus, Integer> statistic(List<Order> orders) {
        int threadCount = Runtime.getRuntime().availableProcessors(); // 按CPU核心数拆分
        int batchSize = (orders.size() + threadCount - 1) / threadCount; // 向上取整

        ExecutorService executor = Executors.newFixedThreadPool(threadCount);
        List<Future<?>> futures = new ArrayList<>();

        // 分片处理
        for (int i = 0; i < threadCount; i++) {
            int start = i * batchSize;
            int end = Math.min((i + 1) * batchSize, orders.size());
            List<Order> subList = orders.subList(start, end);

            futures.add(executor.submit(() -> {
                Map<OrderStatus, Integer> localMap = threadLocal.get();
                for (Order order : subList) {
                    OrderStatus status = order.getStatus();
                    localMap.put(status, localMap.get(status) + 1); // 线程内无锁操作
                }
            }));
        }

        // 等待所有任务完成
        futures.forEach(future -> {
            try {
                future.get();
            } catch (InterruptedException | ExecutionException e) {
                throw new RuntimeException("统计失败", e);
            }
        });
        executor.shutdown();

        // 合并结果
        Map<OrderStatus, Integer> result = new EnumMap<>(OrderStatus.class);
        // 遍历线程池所有线程的ThreadLocal数据（实际需通过线程池获取，此处简化）
        threadLocal.get().forEach((k, v) -> result.merge(k, v, Integer::sum));
        return result;
    }
}
```

**多线程场景核心考点**：

1. **ThreadLocal 优势**：线程内共享变量（如计数器），避免多线程同步锁竞争（比`synchronized`高效）。
2. **任务拆分**：按 CPU 核心数分片（避免过多线程导致上下文切换），子列表无重叠（线程安全）。
3. **结果合并**：最后汇总各线程的局部结果（`merge`方法简化合并逻辑）。
