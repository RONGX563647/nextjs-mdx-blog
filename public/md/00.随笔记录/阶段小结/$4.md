  - 每日花 30 分钟用 “费曼学习法”：合上书，尝试给 “假想面试官” 讲解一个知识点（如 “HashMap 扩容机制”），卡壳处标记为薄弱点。

### 65. JavaWeb 学习策略（视频 vs 笔记）

  - 笔记侧重 “对比与关联”：比如 Servlet vs SpringMVC 控制器的区别、Cookie vs Session 的适用场景，避免孤立记知识点。

### 66. 数据结构与算法补充（黑马课程 + 图码小程序）

### 71. 软件测试全流程（核心阶段与原则）

- **测试前期**：
  - 需求分析：参与评审，输出《需求分析报告》，标注重点测试点与风险（如 “用户权限逻辑是否明确”）。
  - 测试计划：明确范围（功能 / 性能 / 兼容性）、人员分工、时间节点（如 “第 1 周冒烟测试，第 2-3 周系统测试”）、工具选择（如 Jmeter 测性能）。
  - 用例设计：按等价类（有效 / 无效输入）、边界值（如 50MB 文件测 49.9/50/50.1MB）、场景法设计，覆盖正常 / 异常流程。
- **测试执行**：
  - 冒烟测试：验证核心流程（如 “浏览→加购→下单”），不通过则退回开发。
  - 单元 / 集成测试：开发测单元（Junit），测试测模块交互（Postman 接口断言）。
  - 系统测试：覆盖功能（如 “修改密码支持特殊字符”）、性能（Jmeter 模拟 1000 用户登录）、兼容性（多浏览器 / 设备）、安全（OWASP 扫描 SQL 注入）。
  - 回归测试：修复后验证，可选完全回归（全量用例）或选择性回归（仅相关模块）。
- **测试后期**：
  - 缺陷管理：用 Jira 记录（含复现步骤 / 环境 / 截图），按 “新建→确认→修复→验证→关闭” 流转。
  - 测试报告：汇总用例通过率、缺陷分布（按模块 / 严重程度）、质量结论（如 “遗留 3 个一般缺陷，不影响主流程”）。
  - 验收测试：UAT（用户验证）、Alpha（内部）/Beta（外部用户）测试。
- **关键原则**：尽早介入（需求阶段）、全面覆盖（含隐性需求）、持续迭代（敏捷模式同步开发）。

### 72. 手工 AI 测试基础（核心概念与分类）

- **测试核心价值**：发现缺陷、验证需求、提供质量反馈，辅助发布决策。
- **测试分类（高频面试）**：
  - 按阶段：单元测试（函数 / 类，开发主导）→ 集成测试（模块交互，接口为主）→ 系统测试（全量验证）→ 验收测试（用户确认）。
  - 按方法：黑盒（不看代码，测功能）、白盒（看代码，测分支覆盖）、灰盒（接口 + 部分代码逻辑）。
  - 按类型：功能（登录 / 支付）、性能（响应时间≤2 秒）、兼容性（iOS17/Android14 适配）、安全（权限漏洞）。
- **用例设计要素**：编号、模块、步骤、输入数据、预期结果，方法包括等价类（年龄分 0-120 / 外）、边界值（文件大小限制）、错误推测（必填项未填）。
- **缺陷管理要点**：
  - 生命周期：新建→确认→指派→修复→验证→关闭。
  - 核心属性：严重程度（致命 / 严重 / 一般 / 建议）、优先级（高 / 中 / 低）、清晰复现步骤（带环境 + 数据）。

### 73. AI 工具应用（提示词模板与技巧）

- **核心原则**：明确场景（功能 / 性能）、分阶段提问（需求→用例→报告）、人工校验（AI 输出需结合业务）。
- **高频场景 prompt**：
  - 生成用例：“用等价类设计用户注册用例，手机号 11 位，密码 8-20 位含字母 + 数字，输出步骤 + 预期结果”。
  - 缺陷报告：“Chrome 浏览器测试登录，输入正确账号密码无响应，控制台报 Network Error，生成含复现步骤 / 环境 / 严重程度的报告”。
  - 自动化脚本：“用 Python+Selenium 写搜索脚本：打开页面→输入‘测试’→点击→验证结果含‘测试’，加异常处理”。
  - 性能测试：“设计秒杀系统压测方案，500 用户抢购，给 Jmeter 配置建议（线程组 / 断言）”。
- **提效技巧**：带示例提问（“类似登录用例，补充弱网场景”）、限定格式（表格 + 优先级）、追问细化（“加 3 个兼容性测试点”）。

### 74. AI 测试设计（缺陷管理增强）

- **缺陷判断标准**：不符合需求（如登录缺邮箱方式）、违背预期（删除无二次确认）、有技术风险（未处理空指针）。
- **AI 辅助缺陷管理场景**：
  - 报告生成：自动补全环境信息（如 “小米 14/Android14”）、复现步骤（清晰到开发可重现）。
  - 分类优化：AI 标注严重程度（如 “支付卡顿” 标严重，“图标偏移” 标建议）。
  - 趋势分析：周报自动统计模块缺陷占比（如支付 37.5% 最高）、修复率（62.5%）。
  - 风险预测：超时未修复缺陷（如支付接口 4 天）提示延期风险，建议优先处理。
- **避坑点**：不直接提交 AI 报告（漏业务细节），优先处理复杂缺陷（跨模块交互），人工决策优先于 AI 建议。

### 75. 缺陷管理（生命周期与 AI 辅助）

- **缺陷生命周期**：
  1. 新建：测试发现后 10 分钟内提交，含截图 / 日志。
  2. 确认：开发评审是否有效（避免误报）。
  3. 指派 / 修复：开发修复后标 “待验证”。
  4. 验证：测试通过则关闭，否则重开。
- **核心属性**：
  - 严重程度：致命（崩溃）、严重（功能不可用）、一般（界面错位）、建议（体验优化）。
  - 优先级：高（核心业务如支付）、中（次核心）、低（如帮助文档错别字）。
  - 复现步骤：含环境（Chrome120）、数据（账号 186xxxx）、操作（1. 输入...2. 点击...）。
- **AI 辅助场景**：
  - 生成结构化报告（含环境 / 步骤 / 截图指引）。
  - 优化复现步骤（从 “搜索没结果” 细化到 “输入关键词 + 筛选条件 + 无结果截图”）。
  - 预测风险（如超 3 天未修复缺陷，延期概率升 40%）。

### 76. 测试报告（结构与 AI 生成）

- **核心内容**：

  - 概述：项目背景、范围、策略（黑盒 / 自动化占比）。
  - 结果汇总：用例执行数 / 通过率（如 95%）、缺陷统计（按严重程度 / 模块分布）。
  - 缺陷分析：Top 问题（如登录模块 30% 缺陷）、遗留风险（如弱网加载成功率 85%）。
  - 结论：是否通过（如 “致命缺陷归零，同意发布”）、改进建议（开发优化接口限流）。

- **AI 辅助生成**：

  - 自动汇总数据（Jenkins+Allure 数据→生成模块缺陷占比表）。
  - 生成趋势图（折线图展示缺陷收敛）、受众导向报告（给客户侧重用户影响，给开发侧重技术细节）。

- **示例结构**：

  markdown

  ```markdown
  1. 概述（目标/范围）  
  2. 测试结果（用例执行/缺陷分布）  
  3. 重点问题（典型缺陷案例）
  ```

### 78. pytest（测试框架核心）

- **优势**：支持函数 / 类用例（无需继承）、直接用 assert 断言（错误提示详细）、fixtures 机制（灵活管理前置 / 后置）。
- **核心功能**：
  - fixtures：@pytest.fixture 定义前置（如初始化用户数据）→ yield 返回→后置（清理数据），用例通过参数依赖。
  - 参数化：@pytest.mark.parametrize 多组数据（如 [(1+2,3), (3*4,12)]），避免重复代码。
  - 插件：pytest-html（报告）、pytest-xdist（并行执行）、allure-pytest（集成 Allure）。
- **用例发现规则**：文件（test_*.py）、函数（test_*）、类（Test 开头，无__init__）→ 自动识别无需配置。

### 79. Allure（测试报告工具）

- **优势**：可视化展示（通过率 / 缺陷分布图表）、步骤详情（高亮失败步骤）、支持附件（截图 / 日志）、历史趋势对比（跨版本质量波动）。
- **使用流程**：
  1. 安装：allure 命令行工具 + allure-pytest 插件。
  2. 标注用例：@tag（标签）、@severity（优先级）、@step（步骤）、attachment（附件）。
  3. 生成报告：pytest --alluredir = 结果目录→ allure generate 生成报告→ allure open 预览。
- **报告结构**：
  - 概览：总用例数、通过率、趋势图。
  - 用例页：按标签 / 优先级分组，支持搜索。
  - 详情页：步骤、附件、元数据（环境 / 版本）。
- **集成 CI**：Jenkins 流水线配置，自动生成报告并展示趋势。
