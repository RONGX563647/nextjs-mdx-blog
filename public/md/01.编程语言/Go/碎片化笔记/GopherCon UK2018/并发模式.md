## 二、并发基础概念回顾

### 1. 并发（Concurrency） vs 并行（Parallelism）

- **并发**：多个任务 “交替执行”，通过调度让程序**看起来同时运行**（单 CPU 也可实现，核心是 “任务切换”）。
- **并行**：多个任务**真正同时执行**（依赖多 CPU 核心，核心是 “硬件级并行计算”）。
- Go 的优势：原生支持**轻量级并发（goroutine）**，让开发者以 “并发思维” 编码，运行时自动调度实现并行（若硬件支持）。

### 2. Go 并发的核心原语

- **Goroutine**：轻量级 “用户态线程”，创建成本极低（栈初始仅 2KB，可动态扩容），适合大规模并发。
- **Channel**：goroutine 间的通信机制，支持 “同步 / 异步”“有缓冲 / 无缓冲”，是 Go“*通过通信共享内存，而非通过内存共享通信*” 哲学的核心。
- **`sync` 包**：提供传统并发原语（如互斥锁 `Mutex`、读写锁 `RWMutex`、等待组 `WaitGroup`、原子操作 `Atomic` 等），用于精细同步控制。

## 三、经典并发模式解析

### 1. 生产者 - 消费者模式（Producer-Consumer）

- **场景**：生产数据的 goroutine（生产者）与处理数据的 goroutine（消费者）解耦，通过 channel 传递数据。

- **优势**：平衡生产与消费速度，“削峰填谷” 提升系统稳定性。

- 代码示例

  ：

  ```go
  func producer(ch chan<- int) {
      for i := 0; i < 10; i++ {
          ch <- i // 向channel发送数据（生产）
      }
      close(ch) // 生产完毕，关闭channel
  }
  
  func consumer(ch <-chan int) {
      for num := range ch { // 从channel持续读取数据（消费）
          fmt.Println("Consumed:", num)
          time.Sleep(time.Millisecond * 500) // 模拟消费耗时
      }
  }
  
  func main() {
      ch := make(chan int, 3) // 带缓冲channel，平衡生产消费速度
      go producer(ch)
      consumer(ch)
  }
  ```

### 2. Worker Pool（工作池）模式

- **场景**：预先创建固定数量的 goroutine（worker），从 “任务队列” 中获取任务执行，避免频繁创建 / 销毁 goroutine 的开销。

- **优势**：控制并发数（防止资源耗尽）、复用 worker（提升性能）。

- 代码示例

  ：

  go

  ```go
  func worker(id int, tasks <-chan int, results chan<- int) {
      for task := range tasks {
          fmt.Printf("Worker %d processing task: %d\n", id, task)
          result := task * task // 模拟任务计算
          results <- result     // 发送结果
          time.Sleep(time.Millisecond * 300)
      }
  }
  
  func main() {
      tasks := make(chan int, 100)   // 任务队列
      results := make(chan int, 100) // 结果队列
  
      // 创建3个worker
      for i := 1; i <= 3; i++ {
          go worker(i, tasks, results)
      }
  
      // 提交任务
      for i := 0; i < 9; i++ {
          tasks <- i
      }
      close(tasks) // 任务提交完毕
  
      // 收集结果
      for i := 0; i < 9; i++ {
          fmt.Println("Result:", <-results)
      }
  }
  ```

### 3. 扇入（Fan-in）与扇出（Fan-out）模式

- **扇出（Fan-out）**：一个生产者将任务分发到多个 worker**并行处理**（利用多 CPU 核心）。

- **扇入（Fan-in）**：多个生产者的结果，合并到**一个消费者**中处理。

- **场景**：适用于 “高并发任务 + 结果聚合” 的场景（如批量数据并行处理后汇总）。

- 代码示例

  ：

  go

  ```go
  // 扇出：拆分任务到多个worker
  func fanOut(tasks []int, numWorkers int) <-chan int {
      results := make(chan int)
      for i := 0; i < numWorkers; i++ {
          go func(workerID int) {
              // 按worker数量拆分任务，每个worker处理一部分
              for _, task := range tasks[workerID::numWorkers] {
                  results <- task * 2 // 模拟任务处理
              }
          }(i)
      }
      return results
  }
  
  // 扇入：合并多个来源的结果
  func fanIn(resultChs ...<-chan int) <-chan int {
      out := make(chan int)
      var wg sync.WaitGroup
      wg.Add(len(resultChs))
      for _, ch := range resultChs {
          go func(c <-chan int) {
              for res := range c {
                  out <- res
              }
              wg.Done()
          }(ch)
      }
      // 所有resultChs处理完毕后，关闭out
      go func() {
          wg.Wait()
          close(out)
      }()
      return out
  }
  
  func main() {
      tasks := []int{1, 2, 3, 4, 5, 6, 7, 8}
      results := fanOut(tasks, 3) // 3个worker扇出处理
      for res := range results {
          fmt.Println("Final result:", res)
      }
  }
  ```

### 4. 管道（Pipeline）模式

- **场景**：将复杂任务拆分为 “多个阶段”，每个阶段通过 channel 传递数据，形成 “生产→处理→消费” 的流水线。

- **优势**：代码模块化（每个阶段职责单一），易维护、易扩展。

- 代码示例

  （“生成数据→平方计算→结果输出” 流水线）：

  go

  ```go
  // 阶段1：生成数据
  func generate(nums ...int) <-chan int {
      out := make(chan int)
      go func() {
          for _, n := range nums {
              out <- n
          }
          close(out)
      }()
      return out
  }
  
  // 阶段2：处理数据（平方）
  func square(in <-chan int) <-chan int {
      out := make(chan int)
      go func() {
          for n := range in {
              out <- n * n
          }
          close(out)
      }()
      return out
  }
  
  func main() {
      // 组装管道：generate → square → consume
      gen := generate(1, 2, 3, 4)
      squared := square(gen)
  
      // 消费结果
      for n := range squared {
          fmt.Println(n)
      }
  }
  ```

## 四、并发模式的最佳实践与陷阱

### 1. 最佳实践

- **优先用 Channel 通信**：遵循 Go“*通过通信共享内存*” 的哲学，减少互斥锁使用，降低死锁风险。
- **控制 Goroutine 数量**：避免无限制创建 goroutine（如 Worker Pool 模式），防止内存耗尽或调度压力过大。
- **明确 Channel 方向**：使用 “单向 channel”（`chan<-` 只发、`<-chan` 只读），增强代码可读性与类型安全性。
- **及时关闭 Channel**：生产方完成后关闭 channel，消费方可通过`range`或 “`ok` 标识” 判断结束，避免 goroutine 泄漏。

### 2. 常见陷阱

- 死锁（Deadlock）

  ：如 “无缓冲 channel 的发送方与接收方未同时就绪”“循环依赖的 channel 等待”。

  - 示例：`ch := make(chan int); ch <- 1`（无接收方，直接死锁）。

- **资源泄漏（Goroutine Leak）**：goroutine 创建后未正常退出（如 channel 未关闭，导致`range`永久阻塞）。

- **竞态条件（Race Condition）**：必须用共享内存同步时，未正确使用`Mutex`或原子操作，导致数据不一致。

## 五、总结

Go 的并发模式是其核心竞争力之一，通过**goroutine+channel + 同步原语**的组合，可优雅解决复杂并发问题。掌握 “生产者 - 消费者”“Worker Pool”“扇入 / 扇出”“管道” 等模式，并结合最佳实践（规避陷阱），能写出高效、稳定的并发 Go 程序。

# 会议课程笔记：《Go 语言中的并发模式（Concurrency Patterns in Go）》

## 一、并发核心概念与 Go 的设计哲学

### 1. 并发与并行的本质区别

- **并发（Concurrency）**：
  是 “**逻辑上的同时进行**”，通过**任务调度（时间分片）\**实现多个任务交替执行。即使在单 CPU 核心上，也能通过 Go 运行时的快速调度，让多个 goroutine “看起来同时运行”。
  \*Go 视角\*：goroutine 由运行时的\**GPM 调度器**（Goroutine、Processor、Machine）管理，调度开销远低于操作系统线程。
- **并行（Parallelism）**：
  是 “**物理上的同时进行**”，依赖**多 CPU 核心**实现多个任务真正同时执行。
  *Go 视角*：当机器有多个核心时，Go 运行时会自动将 goroutine 调度到不同核心，利用多核实现并行。

### 2. Go 并发原语的设计逻辑

- **Goroutine**：
  - 轻量级：初始栈仅 2KB（可动态扩容至 1GB），创建成本远低于操作系统线程（线程栈通常 MB 级）。
  - 调度高效：**GPM 调度器**针对 IO 密集、CPU 密集任务优化，能智能平衡 goroutine 分布。
- **Channel**：
  - 通信优先：遵循 “*通过通信共享内存，而非通过内存共享通信*”，减少锁的使用。
  - 两种类型：
    - 无缓冲 channel（`make(chan T)`）：发送 / 接收**同步阻塞**，需 “发送方 - 接收方” 同时就绪。
    - 有缓冲 channel（`make(chan T, n)`）：类似 “消息队列”，队列满时发送方阻塞，队列空时接收方阻塞。
- **`sync` 包原语**：
  为 “共享内存” 场景提供工具，如`Mutex`（互斥锁）、`RWMutex`（读写锁）、`WaitGroup`（等待组）、`Atomic`（原子操作）等。

## 二、经典并发模式深度解析

### 1. 生产者 - 消费者模式（Producer-Consumer）

#### 场景与价值

- **场景**：生产与消费速度不匹配时，通过 channel 做 “缓冲层” 解耦逻辑（如日志收集、API 限流）。
- **价值**：削峰填谷，防止速度差异导致系统崩溃。

#### 进阶代码（带错误处理与优雅退出）

go

```go
type Data struct { Value int; Err error }

func producer(ctx context.Context, ch chan<- Data) {
	defer close(ch)
	for i := 0; ; i++ {
		select {
		case <-ctx.Done(): return
		case ch <- Data{Value: i, Err: nil}:
			time.Sleep(200 * time.Millisecond)
		}
	}
}

func consumer(ctx context.Context, ch <-chan Data, wg *sync.WaitGroup) {
	defer wg.Done()
	for data := range ch {
		select {
		case <-ctx.Done(): return
		default:
			if data.Err != nil { /* 处理错误 */ }
			time.Sleep(500 * time.Millisecond)
		}
	}
}

func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
	defer cancel()

	dataCh := make(chan Data, 3)
	var wg sync.WaitGroup
	wg.Add(2)

	go producer(ctx, dataCh)
	for i := 0; i < 2; i++ { go consumer(ctx, dataCh, &wg) }
	
	wg.Wait()
}
```

#### 关键细节

- **Context**：传递 “取消信号”，实现生产者 / 消费者优雅退出。
- **带缓冲 Channel**：平衡生产（200ms / 次）与消费（500ms / 次）的速度差。
- **错误封装**：数据结构嵌入`Err`字段，让消费者处理生产端错误。

### 2. Worker Pool（工作池）模式

#### 场景与价值

- **场景**：需**限制并发数**的任务（如数据库连接池、HTTP 请求限流）。
- **价值**：复用 goroutine，减少创建 / 销毁开销；控制并发数，防止资源耗尽。

#### 进阶代码（带任务优先级、结果回调）

go

```go
type Task struct { ID, Priority int; Do func() (string, error) }
type Result struct { TaskID int; Output string; Err error }

type WorkerPool struct { taskCh, resultCh chan Task; workerNum int; wg sync.WaitGroup }

func NewWorkerPool(workerNum, queueSize int) *WorkerPool {
	return &WorkerPool{
		taskCh:    make(chan Task, queueSize),
		resultCh:  make(chan Result, queueSize),
		workerNum: workerNum,
	}
}

func (wp *WorkerPool) Start(ctx context.Context) {
	for i := 0; i < wp.workerNum; i++ {
		wp.wg.Add(1)
		go func(workerID int) {
			defer wp.wg.Done()
			for {
				select {
				case <-ctx.Done(): return
				case task, ok := <-wp.taskCh:
					if !ok { return }
					output, err := task.Do()
					wp.resultCh <- Result{task.ID, output, err}
				}
			}
		}(i)
	}
}

func main() {
	ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
	defer cancel()

	pool := NewWorkerPool(3, 5)
	pool.Start(ctx)

	// 生成并提交带优先级的任务...
}
```

#### 关键细节

- **任务优先级**：通过排序保证高优先级任务先执行。
- **工作池封装**：对外暴露`Submit`、`Results`等方法，增强可维护性。
- **资源控制**：通过`workerNum`和`queueSize`控制并发与队列压力。

### 3. 扇入（Fan-in）与扇出（Fan-out）模式

#### 场景与价值

- **扇出**：拆分任务到多个 worker**并行处理**，利用多核 CPU（如大文件并行解析）。
- **扇入**：聚合多个任务的结果（如多节点查询结果汇总）。
- **价值**：最大化并行效率，简化结果聚合逻辑。

#### 进阶代码（带超时控制与结果去重）

go

```go
func fanOut(ctx context.Context, tasks []int, numWorkers int) []<-chan int {
	resultChs := make([]<-chan int, numWorkers)
	for i := 0; i < numWorkers; i++ {
		resultCh := make(chan int)
		resultChs[i] = resultCh
		go func(workerID int, ch chan<- int) {
			defer close(ch)
			for j := workerID; j < len(tasks); j += numWorkers {
				select {
				case <-ctx.Done(): return
				default:
					ch <- tasks[j] * tasks[j]
					time.Sleep(300 * time.Millisecond)
				}
			}
		}(i, resultCh)
	}
	return resultChs
}

func fanIn(ctx context.Context, chs ...<-chan int) <-chan int {
	out := make(chan int)
	var wg sync.WaitGroup
	wg.Add(len(chs))
	for _, ch := range chs {
		go func(c <-chan int) {
			defer wg.Done()
			for num := range c {
				select {
				case <-ctx.Done(): return
				case out <- num:
				}
			}
		}(ch)
	}
	go func() { wg.Wait(); close(out) }()
	return out
}
```

#### 关键细节

- **超时控制**：通过`context.WithTimeout`实现整体流程超时。
- **结果去重**：用`map`存储结果，避免重复数据。
- **优雅退出**：所有 goroutine 通过`ctx.Done()`响应取消信号。

### 4. 管道（Pipeline）模式

#### 场景与价值

- **场景**：将复杂任务拆分为**多阶段流水线**（如数据 ETL、Web 请求处理链）。
- **价值**：代码模块化、可复用，各阶段可独立测试 / 替换。

#### 进阶代码（带错误中断与资源清理）

go

```go
type Stage func(context.Context, <-chan int) (<-chan int, error)

func generate(ctx context.Context) (<-chan int, error) { /* 生成数据 */ }
func square(ctx context.Context, in <-chan int) (<-chan int, error) { /* 转换数据 */ }
func outputToFile(ctx context.Context, in <-chan int, file *os.File) (<-chan struct{}, error) { /* 输出数据 */ }

func RunPipeline(ctx context.Context, stages ...Stage) error {
	// 依次执行各阶段，处理错误与资源...
}
```

#### 关键细节

- **阶段错误处理**：某一阶段出错时，通过`return`中断后续流程，实现 “快速失败”。
- **资源清理**：`defer`中关闭资源（如文件），确保流程中断时资源释放。
- **灵活扩展**：新增阶段只需实现`Stage`函数，无需修改核心逻辑。

## 三、并发安全工具与实践

### 1. `sync` 包高级用法

- **`RWMutex` 读多写少优化**：
  读操作允许多个 goroutine 同时进行，写操作独占，降低锁竞争。

  go

  ```go
  var rwMu sync.RWMutex
  var dataMap = make(map[string]string)
  
  func readData(key string) (string, error) {
      rwMu.RLock(); defer rwMu.RUnlock()
      // 读操作...
  }
  
  func writeData(key, val string) {
      rwMu.Lock(); defer rwMu.Unlock()
      // 写操作...
  }
  ```

- **`errgroup.Group` 错误聚合**：
  同时等待多个 goroutine 并捕获第一个错误（扩展库`golang.org/x/sync/errgroup`）。

  go

  ```go
  import "golang.org/x/sync/errgroup"
  
  func fetchURLs(urls ...string) error {
      var eg errgroup.Group
      for _, url := range urls {
          u := url
          eg.Go(func() error {
              // HTTP请求...
          })
      }
      return eg.Wait()
  }
  ```

### 2. `context` 包：控制并发生命周期

- **核心能力**：传递 “取消信号”“超时控制”“元数据”，实现 goroutine 优雅退出。

- 示例（Worker Pool 取消）：

  go

  ```go
  func worker(ctx context.Context, taskCh <-chan Task) {
      for {
          select {
          case <-ctx.Done(): return
          case task, ok := <-taskCh:
              if !ok { return }
              // 处理任务...
          }
      }
  }
  ```

## 四、并发性能调优与常见陷阱

### 1. 性能调优要点

- **减少 Channel 通信开销**：
  对 “高频小数据”，**批量打包**后传递（如每 100 条数据组包）。

  go

  ```go
  // 原方式（每条都通信）
  for i := 0; i < 1000; i++ { ch <- i }
  
  // 优化后（批量打包）
  batch := make([]int, 0, 100)
  for i := 0; i < 1000; i++ {
      batch = append(batch, i)
      if len(batch) == 100 { ch <- batch; batch = batch[:0] }
  }
  ```

- **调整`GOMAXPROCS`**：
  IO 密集型任务可增大`GOMAXPROCS`（让更多 goroutine 同时调度）：

  go

  ```go
  runtime.GOMAXPROCS(runtime.NumCPU() * 2)
  ```

- **控制 CPU 密集型任务的 goroutine 数量**：
  数量接近 CPU 核心数，避免调度开销 > 并行收益。

### 2. 常见陷阱与解决方案

- **死锁（Deadlock）**：
  场景：无缓冲 channel 的 “发送方 - 接收方” 循环等待。
  方案：用**带缓冲 channel**或`select+timeout`避免永久阻塞。
- **Goroutine 泄漏**：
  场景：goroutine 因 channel 未关闭或 context 未取消永久阻塞。
  方案：用`context`传递取消信号，生产端完成后**及时关闭 channel**。
- **竞态条件（Race Condition）**：
  场景：多 goroutine 同时修改共享变量。
  方案：优先用**channel 通信**，或用`Mutex`/`Atomic`保证原子操作。

## 五、总结与延伸学习

### 1. 核心结论

Go 通过`goroutine+channel+同步原语`，高效解决复杂并发问题：

- 生产者 - 消费者：解耦生产与消费。
- Worker Pool：控制并发数，复用资源。
- 扇入 / 扇出：最大化并行效率，聚合结果。
- 管道：模块化拆分任务，提升可维护性。
  结合`context`、`errgroup`等工具，可增强代码健壮性与可观测性。
