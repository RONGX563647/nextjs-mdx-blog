# 斯坦福 CS144 Unit6 预习笔记：网络安全与应用层安全协议

Unit6 是 CS144 从 “基础协议功能” 向 “协议安全性” 的延伸单元，核心聚焦**网络通信的安全威胁**与**应用层安全协议的设计逻辑**，重点围绕 **HTTPS（TLS/SSL 加密）** 展开，同时覆盖 SSH、网络攻击与防御等关键内容。本笔记按 “前置知识衔接→核心安全威胁→安全协议原理→预习重点与实践关联” 结构组织，帮助提前建立 “威胁 - 防御” 的对应思维，为课堂深入学习铺垫。

## 一、前置知识衔接（必回顾）

Unit6 安全内容依赖前序单元的协议基础与密码学入门概念，预习前需巩固以下内容，避免理解断层：

1. **应用层与传输层交互**：HTTP 基于 TCP 传输的 “请求 - 响应” 模型（HTTPS 是 HTTP 与 TLS 的结合，仍依赖 TCP），DNS 基于 UDP 的短报文交互（DNS 安全扩展 DNSSEC 需理解原 DNS 流程）。

2. 基础密码学概念

   （无需深入算法细节，但需明确核心区别）：

   - 对称加密（如 AES）：加密 / 解密用同一密钥，速度快，适合大量数据传输；
   - 非对称加密（如 RSA、ECC）：加密用公钥、解密用私钥，速度慢，适合小数据（如密钥交换）；
   - 哈希函数（如 SHA-256）：将任意数据映射为固定长度哈希值，不可逆，适合完整性校验（如验证数据未被篡改）。

3. **HTTP 的局限性**：HTTP 明文传输（数据可被窃听）、无身份认证（易被冒充）、无完整性校验（数据可被篡改）—— 这些正是 HTTPS 要解决的核心问题。

## 二、Unit6 核心知识点拆解

### 模块 1：网络通信的核心安全威胁

理解 “威胁类型” 是学习安全协议的前提 ——Unit6 首先明确网络中常见的三类安全威胁，所有安全协议的设计均围绕 “防御这三类威胁” 展开：

| 威胁类型              | 定义                                                         | 典型场景                                    | 对 HTTP 的影响                       |
| --------------------- | ------------------------------------------------------------ | ------------------------------------------- | ------------------------------------ |
| 窃听（Eavesdropping） | 攻击者拦截网络中的明文数据，窃取敏感信息（如账号密码、支付信息） | 公共 Wi-Fi 中，攻击者通过抓包获取 HTTP 明文 | 登录密码、表单数据被直接窃取         |
| 篡改（Tampering）     | 攻击者修改传输中的数据（如修改 HTTP 请求参数、响应内容），且不被发现 | 攻击者将 “转账金额 100 元” 改为 “10000 元”  | 数据真实性被破坏，导致业务异常或损失 |
| 冒充（Spoofing）      | 攻击者伪装成合法主体（如伪装成 Web 服务器、客户端），欺骗对方通信 | 攻击者伪造银行官网服务器，骗取用户输入密码  | 身份认证失效，用户向恶意主体泄露信息 |

- **预习关键**：记住 “窃听→需加密”“篡改→需完整性校验”“冒充→需身份认证” 的对应关系 —— 这是后续安全协议（HTTPS、SSH）的设计核心逻辑。

### 模块 2：HTTPS 协议（HTTP Secure）

HTTPS 是 Web 应用的主流安全协议，本质是 “**HTTP + TLS（Transport Layer Security，传输层安全协议，前身是 SSL）**”—— 通过在 HTTP 与 TCP 之间增加 TLS 层，解决 HTTP 的三大安全问题。Unit6 重点覆盖 TLS 1.2/1.3（当前主流版本）的核心机制，需聚焦 “TLS 握手流程”“密钥交换”“身份认证” 三大关键点。

#### 1. HTTPS 与 HTTP 的核心差异

先明确 HTTPS 并非 “新协议”，而是 “HTTP 数据通过 TLS 加密后传输”，两者在端口、传输流程上的差异需先区分：

| 对比维度   | HTTP                                             | HTTPS                                        | 核心安全改进                |
| ---------- | ------------------------------------------------ | -------------------------------------------- | --------------------------- |
| 端口       | 80 端口                                          | 443 端口                                     | 默认使用加密端口，便于识别  |
| 传输流程   | HTTP 报文→TCP 段→IP 包                           | HTTP 报文→TLS 加密→TCP 段→IP 包              | 增加 TLS 加密层，数据非明文 |
| 身份认证   | 无（仅通过 IP / 域名寻址，无法验证服务器合法性） | 基于数字证书验证服务器身份（可选验证客户端） | 防止服务器冒充              |
| 数据完整性 | 无（无法检测数据是否被篡改）                     | 通过 MAC（消息认证码）或哈希校验             | 防止数据篡改                |

#### 2. TLS 层的核心功能（HTTPS 安全的关键）

TLS 层位于 “应用层（HTTP）” 与 “传输层（TCP）” 之间，主要完成三件事：**身份认证、密钥交换、数据加密与完整性校验**，三者协同实现端到端安全：

##### （1）身份认证：基于数字证书（防止冒充）

- **问题**：如何让客户端确信 “正在通信的服务器是合法的（如真正的 Google 服务器，而非攻击者伪造的）”？

- 解决方案

  ：数字证书（由权威 CA 机构签发，如 Let's Encrypt、Symantec），证书包含三大核心信息：

  1. 服务器域名（如 `www.google.com`）；
  2. 服务器的公钥（用于后续密钥交换）；
  3. CA 机构的数字签名（用 CA 私钥对 “域名 + 公钥” 签名，客户端用 CA 公钥验证签名，确认证书未被篡改）。

- 认证流程

  （预习简化版）：

  1. 客户端发起 HTTPS 连接时，服务器向客户端发送数字证书；
  2. 客户端验证证书：① 检查证书是否由 “客户端信任的 CA 机构” 签发（操作系统 / 浏览器预装 CA 公钥）；② 验证 CA 签名是否有效（确认证书未被篡改）；③ 检查证书是否过期、域名是否匹配；
  3. 若证书验证通过，客户端确认服务器身份合法；若失败，浏览器提示 “不安全连接”。

##### （2）密钥交换：协商对称加密密钥（解决 “对称密钥如何安全传输”）

- **问题**：对称加密（如 AES）速度快，适合加密 HTTP 数据，但 “对称密钥” 需客户端与服务器共享 —— 若直接用明文传输密钥，会被窃听；若用非对称加密传输密钥，服务器私钥一旦泄露，所有通信都会被破解。

- 解决方案

  ：TLS 握手阶段通过 “

  非对称加密 + 密钥协商算法

  ” 安全交换对称密钥（称为 “会话密钥”），不同 TLS 版本算法不同，Unit6 重点讲 TLS 1.2 的 RSA 密钥交换和 TLS 1.3 的 ECDHE 密钥交换（更安全，支持前向保密）：

  - RSA 密钥交换

    （TLS 1.2 常用）：

    1. 客户端生成 “预主密钥（Pre-Master Secret，随机数）”，用服务器证书中的公钥加密后发给服务器；
    2. 服务器用自身私钥解密，得到预主密钥；
    3. 客户端与服务器分别用 “预主密钥 + 握手过程中的随机数”，通过相同算法生成 “会话密钥”（对称密钥）；

  - ECDHE 密钥交换

    （TLS 1.3 推荐，支持前向保密）：

    1. 客户端与服务器各自生成 “临时 ECC 密钥对”（公钥 + 私钥），互相发送临时公钥；
    2. 双方用对方的临时公钥和自身的临时私钥，计算出相同的 “共享密钥”；
    3. 基于共享密钥生成会话密钥（即使服务器私钥泄露，已加密的历史数据也无法解密，即 “前向保密”）。

##### （3）数据加密与完整性校验（防止窃听与篡改）

- **数据加密**：TLS 握手完成后，客户端与服务器用 “会话密钥”（对称加密，如 AES-256-GCM）加密 HTTP 报文（请求 / 响应均加密），攻击者即使拦截数据，也无法解密（无会话密钥）。
- **完整性校验**：同时使用 “加密 + 消息认证码（MAC）” 或 “认证加密算法（AEAD，如 GCM 模式）”—— 加密的同时生成 “校验值”，接收方解密后验证校验值，若数据被篡改，校验值不匹配，直接丢弃数据。

#### 3. TLS 握手流程（预习核心，简化版）

TLS 握手是 HTTPS 建立安全连接的关键步骤，Unit6 会详细讲解，预习时需记住 “四步核心交互”（以 TLS 1.2 RSA 密钥交换为例）：

1. **客户端 Hello**：客户端向服务器发送：① 支持的 TLS 版本（如 TLS 1.2）；② 支持的加密套件（如 AES-256-CBC-SHA256）；③ 客户端随机数（用于后续生成会话密钥）。
2. **服务器 Hello + 证书 + Hello Done**：服务器回应：① 选定的 TLS 版本和加密套件；② 服务器随机数；③ 服务器数字证书（含公钥）；④ 发送 “Hello Done”，表示服务器 Hello 阶段结束。
3. **客户端密钥交换 + Change Cipher Spec+Finished**：客户端操作：① 验证服务器证书通过后，生成 “预主密钥”，用服务器公钥加密后发送；② 发送 “Change Cipher Spec”，告知服务器 “后续数据将用会话密钥加密”；③ 用 “客户端随机数 + 服务器随机数 + 预主密钥” 生成会话密钥，加密 “Finished” 消息（含握手过程的哈希值）发送，验证握手未被篡改。
4. **服务器 Change Cipher Spec+Finished**：服务器操作：① 用私钥解密预主密钥，生成会话密钥；② 发送 “Change Cipher Spec”；③ 加密 “Finished” 消息发送；④ 客户端验证服务器的 “Finished” 消息通过后，TLS 握手完成，后续 HTTP 数据均通过会话密钥加密传输。

### 模块 3：其他核心安全协议与防御手段

Unit6 除 HTTPS 外，还会简要覆盖 SSH、DNSSEC 及网络层防御手段，重点理解 “不同场景下的安全需求差异”：

#### 1. SSH 协议（Secure Shell）

- **应用场景**：远程登录服务器（替代明文的 Telnet 协议）、远程执行命令、文件传输（SCP/SFTP 基于 SSH）。

- 核心安全机制

  ：

  - 身份认证：支持 “密码认证” 或 “公钥认证”（更安全，客户端将公钥存到服务器，登录时用私钥签名验证身份）；
  - 数据加密：基于对称加密（如 AES）传输数据，密钥通过非对称加密（如 RSA）交换；
  - 完整性校验：用 MAC 或哈希校验数据，防止篡改。

- **与 HTTPS 的差异**：HTTPS 聚焦 “Web 应用的请求 - 响应安全”，SSH 聚焦 “远程控制与文件传输安全”，但核心安全逻辑一致（身份认证 + 密钥交换 + 加密）。

#### 2. DNSSEC（DNS 安全扩展）

- **问题**：传统 DNS 易被 “DNS 劫持”（攻击者篡改 DNS 解析结果，将 `www.google.com` 解析到恶意 IP），且无身份认证（无法确认 DNS 响应是否来自合法服务器）。

- 解决方案

  ：DNSSEC 通过 “数字签名” 验证 DNS 记录的完整性与合法性：

  - 权威 DNS 服务器用私钥对 DNS 记录（如 A 记录）签名，生成 “RRSIG 记录”（资源记录签名）；
  - 客户端获取 DNS 响应时，用权威服务器的公钥验证 RRSIG 记录，确认 DNS 记录未被篡改、来源合法。

#### 3. 网络层防御手段

- **防火墙（Firewall）**：位于网络边界（如家庭路由器、企业网关），按 “规则” 过滤数据包（如禁止外部访问内部 22 端口（SSH）、只允许 80/443 端口的 Web 流量），防御外部攻击。
- **入侵检测 / 防御系统（IDS/IPS）**：IDS 监控网络流量，检测异常行为（如频繁尝试 SSH 登录）并告警；IPS 在 IDS 基础上，可主动阻断攻击流量（如拉黑恶意 IP）。

## 四、预习实践关联与资源推荐

### 1. 实践要点（为实验做准备）

CS144 Unit6 可能配套 “TLS 握手分析” 或 “简易 HTTPS 客户端” 实验，预习时可提前用工具感受安全协议：

- 用 Wireshark 抓包分析 HTTPS

  ：

  1. 打开 Wireshark，过滤规则设为 `tcp.port == 443`；
  2. 浏览器访问 `https://www.example.com`，抓取 TLS 握手包；
  3. 观察 “Client Hello”“Server Hello”“Certificate”“Finished” 等包的内容，对应预习的握手流程。

- 用 OpenSSL 工具验证证书

  ：

  - 执行命令 `openssl s_client -connect www.example.com:443`，查看服务器证书信息（如 issuer（CA 机构）、validity（有效期）、public key（公钥）），理解证书的结构。

### 2. 预习资源推荐

1. **官方教材**：《计算机网络：自顶向下方法》（第 8 章 “计算机网络安全”）—— 对应 Unit6 核心威胁与安全协议，讲解通俗易懂。
2. **TLS 规范文档**：[RFC 8446](https://datatracker.ietf.org/doc/html/rfc8446)（TLS 1.3 规范，重点看 “Handshake Protocol” 章节，理解握手细节）。
3. **工具文档**：OpenSSL 官方文档（[OpenSSL Wiki](https://wiki.openssl.org/)）—— 学习如何用命令行工具操作 TLS 与证书，为实验打基础。

通过以上预习，可建立 “威胁 - 防御 - 协议” 的对应框架，明确 HTTPS/TLS 等安全协议的设计初衷与核心逻辑。重点关注 “TLS 如何用非对称加密解决对称密钥的安全传输”“数字证书如何验证身份”，这些是 Unit6 课堂深入的核心，也是后续实践的关键。
