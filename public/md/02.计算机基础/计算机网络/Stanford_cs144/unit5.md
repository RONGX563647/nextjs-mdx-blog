### 一、DNS（域名系统）

#### 1. 资源记录（Resource Record, RR）格式

每个 DNS 响应包含多条 RR，格式为：

plaintext











```plaintext
域名（Name） | 类型（Type） | 类（Class，通常IN） | 生存时间（TTL） | 数据长度（RDLength） | 数据（RData）
```

- Type 字段

  ：

  - `A`：RData 为 IPv4 地址（4 字节）；
  - `AAAA`：RData 为 IPv6 地址（16 字节）；
  - `CNAME`：RData 为目标域名；
  - `NS`：RData 为负责该域的 DNS 服务器域名；
  - `MX`：RData 包含优先级（数字越小优先级越高）和邮件服务器域名。

#### 2. 递归与迭代的细节流程

- **递归查询**：客户端 → 本地 DNS（全程代理查询）。本地 DNS 先查根服务器（获取顶级域服务器地址）→ 查顶级域服务器（获取二级域服务器地址）→ 查二级域服务器（获取目标 IP），最终将 IP 返回客户端。
- **迭代查询**：若本地 DNS 无缓存，会**依次向根、顶级域、二级域服务器发送迭代请求**。例如，根服务器不会直接返回 IP，而是告诉本地 DNS：“去问`.com`顶级域服务器”；顶级域服务器再告诉本地 DNS：“去问`example.com`的权威服务器”，直到拿到 IP。

#### 3. 缓存与 TTL 机制

DNS 服务器（包括本地 DNS）会缓存查询结果，**TTL（生存时间）** 决定缓存有效期。若缓存过期，需重新查询。这既减少网络请求，也可能导致 “域名更新后，部分用户仍访问旧 IP”（需等缓存失效）。

#### 4. DNSSEC（DNS 安全扩展）

解决 DNS “缓存投毒”“数据篡改” 问题。通过**数字签名**验证 DNS 响应的真实性：权威服务器为 RR 签名，递归服务器验证签名，确保数据未被篡改。

### 二、NAT（网络地址转换）

#### 1. NAPT 的端口映射表

NAPT 设备维护 “内网 IP: 端口 → 公网 IP: 端口” 的映射表，示例：

| 内网信息          | 公网信息          | 协议 |
| ----------------- | ----------------- | ---- |
| 192.168.1.2:50000 | 203.0.113.1:60000 | TCP  |
| 192.168.1.3:50001 | 203.0.113.1:60001 | UDP  |

当外网响应`203.0.113.1:60000`时，NAT 设备根据映射表，转发到`192.168.1.2:50000`。

#### 2. NAT 穿透技术（解决 P2P 通信问题）

NAT 破坏了 “端到端” 通信，导致 P2P（如视频通话、BT 下载）难以直接建立连接。常见解决方案：

- **STUN（Session Traversal Utilities for NAT）**：客户端向 STUN 服务器发请求，获取 “NAT 分配的公网 IP: 端口”，然后告知对方，尝试直连。
- **TURN（Traversal Using Relays around NAT）**：若 STUN 失败（如对称型 NAT），则通过 TURN 服务器**中继转发**数据（但会增加延迟和服务器负载）。
- **ICE（Interactive Connectivity Establishment）**：整合 STUN、TURN，自动选择最优连接方式。

#### 3. NAT 对传输层的影响

- **TCP**：NAT 需维护 TCP 连接的 “状态”（如 SYN、ESTABLISHED），确保握手、数据传输、断开的正确性。
- **UDP**：因 UDP 无连接，NAT 需通过 “空闲超时” 回收端口映射（如一段时间无 UDP 包，映射表项被删除）。

### 三、HTTP（超文本传输协议）

#### 1. 请求与响应头部字段（部分关键）

- 请求头

  ：

  - `Host`：指定目标域名（HTTP/1.1 必需，用于虚拟主机多域名共享 IP）；
  - `User-Agent`：客户端类型（浏览器、爬虫等）；
  - `Cookie`：携带会话信息；
  - `Connection: Keep-Alive`：请求持久连接。

- 响应头

  ：

  - `Content-Type`：数据类型（如`text/html`、`application/json`）；
  - `Set-Cookie`：服务端设置 Cookie；
  - `Cache-Control`：缓存策略（如`max-age=3600`表示缓存 1 小时）；
  - `Transfer-Encoding: chunked`：分块传输（无需提前指定 Content-Length）。

#### 2. HTTP/2 的二进制帧结构

HTTP/2 将请求 / 响应拆分为**帧（Frame）**，帧格式包含：

- 长度（Length）、类型（Type，如数据帧、头部帧）、标志（Flags）、流标识（Stream ID，区分不同请求 / 响应的 “流”）、负载（Payload）。

**多路复用**的核心：不同流的帧可在同一 TCP 连接中**乱序传输**，接收端再按 Stream ID 重组，彻底解决 HTTP/1.1 的 “队头阻塞”。

#### 3. HTTP/3 与 QUIC 协议

HTTP/3 基于**QUIC（UDP 上的可靠传输协议）**，解决 TCP+TLS 的延迟问题：

- **0-RTT 连接建立**：首次连接后，后续可跳过 TLS 握手，直接传数据；
- **内置拥塞控制**：QUIC 自定义拥塞控制算法，不依赖操作系统的 TCP 栈；
- **流级别的拥塞控制**：某条流拥塞，不影响其他流（TCP 是连接级拥塞控制，一个流卡壳全连接卡壳）。

### 四、BitTorrent（BT 下载）

#### 1. .torrent 种子文件结构

包含：

- `announce`：Tracker 服务器地址；
- `info`：文件元数据（文件名、长度、分块大小、分块的 SHA-1 哈希值）；
- 分块哈希用于**验证数据完整性**：下载某分块后，计算哈希与种子里的比对，不一致则重新下载。

#### 2. 分块选择策略

- **最稀有优先（Rarest First）**：优先下载 “全网拥有者最少” 的分块，加速分块在 Peer 间的传播，提升整体下载效率。
- **结束游戏（End Game）**：下载接近完成时，同时向多个 Peer 请求剩余的少量分块，减少最后几个分块的等待时间。

#### 3. Tracker 协议与 DHT

- **Tracker 协议**：客户端向 Tracker 发`GET /announce?info_hash=...&peer_id=...&port=...`请求，Tracker 返回当前拥有该种子的 Peer 列表。
- **DHT（分布式哈希表）**：基于**Kademlia 算法**，去中心化存储 Peer 信息。即使 Tracker 不可用，客户端也能通过 DHT 网络发现其他 Peer。

### 五、DHCP（动态主机配置协议）

#### 1. DHCP 报文格式（部分选项）

DHCP 报文是**UDP 报文**（客户端端口 68，服务器端口 67），包含：

- `op`：操作码（1 = 请求，2 = 响应）；

- `yiaddr`：服务器分配的 IP（仅响应报文有效）；

- ```
  options
  ```

  ：可选字段，如：

  - 选项 53（消息类型，如 Discover/Offer/Request/Ack）；
  - 选项 1（子网掩码）；
  - 选项 3（默认网关）；
  - 选项 6（DNS 服务器地址）；
  - 选项 51（IP 租约时间）。

#### 2. 租约管理细节

- 租约到期前**50% 时间**，客户端发`DHCPREQUEST`续约；若成功，租约延长至原时长。
- 若续约失败，租约到期前**87.5% 时间**，客户端再次尝试续约；若仍失败，租约到期后，客户端需重新发起 DHCP 发现流程。

#### 3. DHCPv6（针对 IPv6 的改进）

- 基于**ICMPv6**（不再用 UDP），报文格式和流程更简洁。
- 支持 “有状态”（类似 DHCPv4，集中分配 IP）和 “无状态”（通过路由公告 RA 分配前缀，主机自行生成 IP）两种模式。

这些细节覆盖了协议的**字段定义、算法逻辑、边缘 case 处理**，能更深入理解技术的底层运行机制

## 一、前置知识衔接（必回顾）

Unit5 依赖前序单元的核心概念，预习前需确保掌握以下内容，避免理解断层：

1. **传输层服务差异**：TCP（可靠、面向连接、字节流）与 UDP（不可靠、无连接、数据报）的适用场景（应用层协议需选择合适的传输层协议，如 HTTP 用 TCP，DNS 用 UDP）。
2. **端口机制**：传输层通过端口号区分同一主机上的不同应用（如 HTTP 用 80 端口，HTTPS 用 443 端口，DNS 用 53 端口），应用层协议需绑定固定端口以实现 “客户端寻址”。
3. **分层封装逻辑**：应用层数据 → 传输层头部（TCP/UDP）→ 网络层头部（IP）→ 链路层头部（以太网），Unit5 聚焦 “应用层数据的格式与交互规则”。

## 二、Unit5 核心知识点拆解

### 模块 1：应用层概述与核心架构

#### 1. 应用层的核心定位

- **功能**：为用户提供具体的网络服务（如网页浏览、文件传输、域名解析），是 “用户能直接感知” 的网络层级。
- **本质**：定义 “应用进程间的通信规则”—— 包括数据格式（如 HTTP 请求 / 响应报文）、交互流程（如 HTTP 的 “请求 - 响应” 循环）、错误处理（如 DNS 的查询失败重试）。
- **依赖下层**：应用层不直接操作硬件，而是通过**传输层接口（如 Socket）** 调用 TCP/UDP 服务，间接使用网络层、链路层的功能。

#### 2. 两种核心应用架构

CS144 重点讲解 “客户端 - 服务器（C/S）” 和 “对等网络（P2P）” 架构，两者的差异是预习重点：

| 架构类型               | 核心特征                                                     | 优势                                 | 劣势                               | 典型应用                                  |
| ---------------------- | ------------------------------------------------------------ | ------------------------------------ | ---------------------------------- | ----------------------------------------- |
| 客户端 - 服务器（C/S） | 存在 “中心化服务器”，客户端主动请求，服务器被动响应（如浏览器→Web 服务器） | 服务稳定（服务器可控）、数据一致性高 | 服务器压力大（单点瓶颈）、扩展性差 | HTTP（网页）、FTP（文件传输）             |
| 对等网络（P2P）        | 无中心化节点，每个节点既是客户端也是服务器（如迅雷下载）     | 扩展性强（节点越多带宽越高）、成本低 | 节点可靠性低（易离线）、数据同步难 | BitTorrent（文件共享）、Skype（视频通话） |

- **预习关键**：理解 C/S 架构的 “请求 - 响应” 模型（后续 HTTP 基于此），以及 P2P 架构如何通过 “节点发现”（如通过 tracker 服务器）解决 “去中心化寻址” 问题。

### 模块 2：HTTP 协议（HyperText Transfer Protocol）

HTTP 是 Web 应用的核心协议，Unit5 重点覆盖 HTTP/1.1（当前主流）和 HTTP/2 的核心特性，需聚焦 “报文格式”“交互流程”“无状态设计” 三大关键点。

#### 1. HTTP 的核心设计原则

- **基于 TCP 传输**：HTTP 依赖 TCP 的可靠性（确保请求 / 响应不丢失），但需先建立 TCP 连接（三次握手），早期 HTTP/1.0 为 “短连接”（一次请求对应一次 TCP 连接），HTTP/1.1 引入 “长连接”（`Connection: keep-alive`）减少连接开销。
- **无状态协议**：服务器不存储客户端的历史交互信息（如 “是否登录过”），每次请求都是独立的 —— 优势是服务器压力小、扩展性强；劣势是需通过 “Cookie/Session” 补充状态（如登录状态维持）。
- **请求 - 响应模型**：客户端发送 “请求报文”，服务器返回 “响应报文”，一次交互完成后可保持 TCP 连接（长连接）以处理后续请求。

#### 2. HTTP 报文格式（预习重点）

HTTP 报文由 “起始行 + 首部字段 + 空行 + 实体主体” 四部分组成，请求报文与响应报文的格式有差异，需准确区分：

##### （1）请求报文格式

http











```http
<方法> <请求URI> <HTTP版本>  # 起始行（如 GET /index.html HTTP/1.1）
<首部字段名1>: <值1>        # 首部字段（如 Host: www.example.com）
<首部字段名2>: <值2>
...
（空行，分隔首部与实体主体）
<实体主体>                  # 可选（如 POST 请求的表单数据）
```

- 核心字段解析

  ：

  - 方法（Method）：表示请求的 “操作类型”，Unit5 重点掌握 4 种：
    - `GET`：获取资源（如浏览网页，实体主体为空，参数拼在 URI 后，如 `?id=1`）；
    - `POST`：提交资源（如表单提交，参数放在实体主体中，更安全）；
    - `HEAD`：仅获取响应首部（不返回实体主体，用于检查资源是否存在）；
    - `OPTIONS`：查询服务器支持的 HTTP 方法（如跨域请求预检）。
  - Host 首部：指定目标服务器的域名（HTTP/1.1 强制要求，因一台服务器可能托管多个域名，如 nginx 虚拟主机）。

##### （2）响应报文格式

http











```http
<HTTP版本> <状态码> <原因短语>  # 起始行（如 HTTP/1.1 200 OK）
<首部字段名1>: <值1>            # 首部字段（如 Content-Type: text/html）
<首部字段名2>: <值2>
...
（空行，分隔首部与实体主体）
<实体主体>                      # 服务器返回的资源（如 HTML 代码、图片二进制数据）
```

- 核心字段解析

  ：

  - 状态码（Status Code）：表示请求的处理结果，分 5 类，需记住常用码：
    - 2xx（成功）：200 OK（请求成功）；
    - 3xx（重定向）：302 Found（临时重定向）、304 Not Modified（资源未修改，使用缓存）；
    - 4xx（客户端错误）：400 Bad Request（请求格式错误）、404 Not Found（资源不存在）、403 Forbidden（权限不足）；
    - 5xx（服务器错误）：500 Internal Server Error（服务器内部错误）、503 Service Unavailable（服务器过载）。
  - Content-Type 首部：指定实体主体的 “数据类型”（如 `text/html` 表示 HTML 文本，`image/jpeg` 表示 JPEG 图片），客户端据此解析数据。

#### 3. HTTP/1.1 vs HTTP/2（预习难点）

Unit5 会对比两者的差异，核心改进聚焦 “性能优化”，需理解 HTTP/1.1 的瓶颈与 HTTP/2 的解决方案：

| 对比维度   | HTTP/1.1                                                     | HTTP/2                                                       | 核心改进目的                  |
| ---------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ----------------------------- |
| 连接复用   | 长连接（可处理多请求），但 “串行请求”（同一连接中，前一个请求完成后才能发下一个） | 多路复用（同一连接中，多个请求通过 “流（Stream）” 并行传输，互不阻塞） | 解决 “队头阻塞”，提升并发效率 |
| 数据压缩   | 仅实体主体可压缩，首部字段不压缩（重复首部占带宽，如 Host、Cookie） | 首部压缩（HPACK 算法，共享首部字典，减少重复字段传输）       | 减少带宽消耗                  |
| 服务器推送 | 仅支持 “客户端请求→服务器响应”，无法主动推送资源             | 服务器推送（如请求 HTML 时，主动推送关联的 CSS/JS，无需客户端再请求） | 减少请求次数，降低延迟        |

### 模块 3：DNS 协议（Domain Name System）

DNS 是 “域名→IP 地址” 的解析协议，解决 “人类易记域名” 与 “机器识别 IP” 的映射问题，Unit5 重点覆盖其 “分层解析架构” 与 “查询流程”。

#### 1. DNS 的核心功能与传输层选择

- **功能**：将域名（如 `www.google.com`）解析为 IPv4/IPv6 地址（如 `142.250.74.142`），同时支持反向解析（IP→域名）、邮件服务器定位（MX 记录）等。
- **基于 UDP 传输**：DNS 查询通常数据量小（<512 字节），UDP 低延迟的优势更明显；若查询结果超过 512 字节（如返回多个 IP），会自动切换为 TCP 传输。
- **端口**：DNS 服务器默认使用 53 端口（UDP/TCP 均支持）。

#### 2. DNS 的分层解析架构（预习重点）

DNS 采用 “分布式分层架构”，避免单点故障，同时提升解析效率，从顶层到下层分为 4 级（实际可能更多）：

1. **根域名服务器（Root Server）**：全球共 13 组（通过任播技术分布），存储顶级域名服务器的地址（如 `.com`、`.org` 服务器），是解析的 “入口”。
2. **顶级域名服务器（TLD Server）**：负责顶级域名（如 `.com`、`.cn`、`.edu`）的解析，存储二级域名服务器的地址（如 `google.com` 服务器）。
3. **权威域名服务器（Authoritative Server）**：负责特定域名（如 `google.com`）的解析，存储该域名的具体 DNS 记录（如 A 记录、AAAA 记录），是域名解析的 “最终来源”。
4. **本地域名服务器（Local Server）**：用户设备（如电脑、手机）配置的 DNS 服务器（如路由器分配的 `192.168.1.1`，或运营商的 `223.5.5.5`），负责缓存解析结果，减少重复查询。

#### 3. DNS 查询流程（递归 + 迭代）

以 “用户查询 `www.google.com`” 为例，解析流程分两步，需理解 “递归查询”（用户→本地服务器）与 “迭代查询”（本地服务器→根 / 顶级 / 权威服务器）的区别：

1. **用户→本地 DNS 服务器（递归查询）**：用户设备向本地服务器发送查询请求，本地服务器承诺 “要么返回结果，要么报错”，用户无需关心后续步骤。

2. 本地 DNS 服务器→各级服务器（迭代查询）

   ：

   - 本地服务器先查缓存，若有结果直接返回；若无，向 “根域名服务器” 查询 `.com` 顶级域名服务器的地址。
   - 根服务器返回 `.com` 顶级域名服务器的地址，本地服务器向其查询 `google.com` 权威服务器的地址。
   - 顶级域名服务器返回 `google.com` 权威服务器的地址，本地服务器向其查询 `www.google.com` 的 IP 地址。
   - 权威服务器返回 `www.google.com` 的 A 记录（IPv4 地址），本地服务器缓存该结果后，返回给用户。

#### 4. 常见 DNS 记录类型

DNS 服务器存储 “记录（Record）”，不同记录对应不同功能，Unit5 需掌握 4 种核心记录：

- **A 记录**：域名→IPv4 地址（如 `www.google.com. 300 IN A 142.250.74.142`，`300` 为 TTL，缓存时间，单位秒）；
- **AAAA 记录**：域名→IPv6 地址（对应 IPv6 解析）；
- **CNAME 记录**：域名→另一个域名（别名，如 `blog.google.com` 指向 `googleblog.blogspot.com`）；
- **MX 记录**：域名→邮件服务器地址（如 `google.com` 的 MX 记录指向 `alt1.aspmx.l.google.com`）。

# 知识点

这些内容属于斯坦福 CS144 中 ** 应用层协议与网络地址转换（NAT）** 的核心知识点，涵盖 DNS、NAT、HTTP、BitTorrent、DHCP 等关键技术，以下逐一讲解：

### 一、DNS（域名系统，Domain Name System）

**核心作用**：将**人类易记的域名**（如`www.example.com`）映射为**计算机网络的 IP 地址**，解决 “域名→IP” 的解析问题，是互联网的 “地址簿”。

#### 1. 分层架构与解析流程

- **层次化命名空间**：分为**根域（Root）**、**顶级域（TLD，如.com、.org、.cn）**、**二级域（如[example.com](https://example.com/)）**、**主机名（如[www.example.com](https://www.example.com/)）**。各级由不同 DNS 服务器负责解析。

- 解析方式

  ：

  - **递归查询**：客户端向 “本地 DNS 服务器” 发送请求，由本地服务器**全程递归查询**所有层级（根→顶级域→二级域），最终返回 IP 给客户端。
  - **迭代查询**：若本地服务器无法直接解析，会**逐步向根、顶级域、二级域服务器迭代请求**，直到获取 IP，再返回给客户端。

#### 2. 关键记录类型

- **A 记录**：将域名映射到**IPv4 地址**。
- **AAAA 记录**：将域名映射到**IPv6 地址**。
- **CNAME 记录**：“别名记录”，将域名映射到**另一个域名**（如`www.example.com` → `example.com`）。
- **MX 记录**：“邮件交换记录”，指定处理域名邮件的服务器。

### 二、NAT（网络地址转换，Network Address Translation）

**核心作用**：解决**IPv4 地址短缺**问题，将内网**私有 IP**转换为外网**公有 IP**，实现多设备共享公网 IP 上网。

#### 1. NAT 类型

- **静态 NAT**：内网私有 IP 与公网 IP**一对一静态映射**（多用于需要固定公网地址的服务器）。
- **动态 NAT**：内网私有 IP 从**公网 IP 池**中**动态选择**一个映射（有地址池耗尽风险）。
- **NAPT（PAT，端口地址转换）**：最常用！不仅转换 IP，还**转换端口号**，实现**多个内网 IP 共享一个公网 IP**（如家庭路由器：通过端口区分不同设备的网络连接）。

#### 2. 工作机制

- 内网设备发往外网的数据包：NAT 设备修改 “源 IP” 为自身公网 IP，并记录`（内网IP:源端口）→（公网IP:映射端口）`的映射表。
- 外网返回的数据包：NAT 设备根据映射表，将 “目标 IP: 端口” 转换为内网对应的 “IP: 端口”，转发给内网设备。

#### 3. 影响与局限性

- 解决 IPv4 地址短缺，但**破坏了 IP 的 “端到端透明性”**，导致 P2P、VoIP 等应用需要 “NAT 穿透” 技术（如 STUN、TURN）。
- 增加网络延迟（NAT 设备需处理数据包），且不利于故障排查（IP 被转换，网络路径不透明）。

### 三、HTTP（超文本传输协议，Hypertext Transfer Protocol）

**核心作用**：Web 浏览器与服务器之间传输超媒体内容（网页、图片、视频等）的应用层协议。

#### 1. 版本演进与特性

- **HTTP/1.0**：无状态、**短连接**（每次请求都要新建 TCP 连接，效率低）。

- HTTP/1.1

  ：

  - **持久连接（Keep-Alive）**：默认开启，同一 TCP 连接可处理**多个请求**，减少连接建立开销。
  - **管道化（Pipelining）**：客户端可在未收到前一个响应时，发送多个请求（但因 “队头阻塞”，实际效果有限）。
  - **分块传输编码**：支持 “动态生成内容” 的流式传输（无需提前知道内容总长度）。

- HTTP/2

  ：

  - **二进制帧**：将请求 / 响应拆分为二进制帧，解析更高效。
  - **多路复用**：同一 TCP 连接上，**多个请求 / 响应可并行传输**（通过 “流标识” 区分），彻底解决 “队头阻塞”。
  - **头部压缩（HPACK）**：减少重复 HTTP 头部的传输量。

- **HTTP/3**：基于**QUIC（UDP 协议）**，进一步优化延迟（避免 TCP 的队头阻塞），支持 “0-RTT” 快速连接建立。

#### 2. 请求 - 响应模型

- **请求报文**：包含「方法（GET/POST 等）、URL、HTTP 版本、头部字段（如 Host、Cookie）、请求体（如 POST 的表单数据）」。
- **响应报文**：包含「HTTP 版本、状态码（200 OK/404 Not Found 等）、头部字段（如 Content-Type）、响应体（如网页内容）」。

#### 3. 无状态与 Cookie

- HTTP 本身**无状态**（无法记住用户历史操作）。
- **Cookie**：通过在客户端存储小文本文件，服务器用`Set-Cookie`头部设置，客户端后续请求携带 Cookie，实现 “会话跟踪”（如登录状态保持）。

### 四、BitTorrent（BT 下载）

**核心作用**：一种**P2P（点对点）文件共享协议**，用于高效分发大文件（如电影、软件）。

#### 1. 核心概念

- **种子（Torrent）**：包含文件元数据（文件名、分块信息、Tracker 地址）的小文件。
- **分块（Piece）**：大文件被分割为多个小分块（如 256KB），下载时不同设备（Peer）交换不同分块，实现**并行下载**。
- **Tracker**：跟踪 “哪些 Peer 拥有文件分块”，为客户端提供 Peer 列表。
- **DHT（分布式哈希表）**：去中心化的 Peer 发现方式（无需 Tracker，通过分布式网络存储 Peer 信息）。

#### 2. 工作原理

- 客户端（Peer）同时是 “下载者” 和 “上传者”：下载时从多个 Peer 获取不同分块，同时向其他 Peer 上传已下载的分块。
- **激励机制**：采用 “tit-for-tat（以牙还牙）” 策略 —— 优先向 “上传速度快” 的 Peer 请求数据，同时也为这些 Peer 提供更快的上传。

### 五、DHCP（动态主机配置协议，Dynamic Host Configuration Protocol）

**核心作用**：自动为网络设备分配**IP 地址、子网掩码、网关、DNS 服务器**等网络配置参数，简化设备接入网络的流程。

#### 1. 工作流程（DORA）

- **发现（Discover）**：新设备广播`DHCPDISCOVER`报文，寻找 DHCP 服务器。
- **提供（Offer）**：DHCP 服务器广播`DHCPOFFER`报文，提供 “可用 IP + 配置参数”。
- **请求（Request）**：设备选择一个`DHCPOFFER`（通常选第一个收到的），广播`DHCPREQUEST`报文请求该 IP。
- **确认（Acknowledge）**：DHCP 服务器广播`DHCPACK`报文，确认 IP 分配，设备即可使用该 IP。

#### 2. 租约管理

- 分配的 IP 有**租期（Lease Time）**：到期前设备需发`DHCPREQUEST`续约；若不续约，IP 会被回收。
- 设备可主动发`DHCPRELEASE`报文释放 IP。

### 总结

这些技术共同支撑互联网的**应用层服务**与**网络资源管理**：

- DNS 是 “互联网地址簿”，解决域名→IP 的映射；
- NAT 缓解 IPv4 地址短缺，实现内网多设备共享公网 IP；
- HTTP 是 Web 的基础协议，经历多版本优化传输效率；
- BitTorrent 是典型 P2P 应用，高效分发大文件；
- DHCP 自动化网络配置，简化设备入网流程。
