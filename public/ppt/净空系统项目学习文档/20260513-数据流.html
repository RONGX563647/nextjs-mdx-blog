2.2 MQTT 协议通信

MQTT（Message Queuing Telemetry Transport） 是一种轻量级的消息发布/订阅传输协议，专为物联网场景设计。本项目使用 MQTT 作为设备与后端之间的通信协议。

📖 MQTT 核心概念
  • Publish（发布）：设备向特定 Topic 发送消息
  • Subscribe（订阅）：后端订阅特定 Topic 接收消息
  • Topic（主题）：消息的分类路径，如 ipc/device001/data/target
  • QoS（服务质量）：本项目使用 QoS 1（至少送达一次）
  • Broker（代理）：MQTT 消息服务器，本项目使用 Eclipse Mosquitto

MQTT 连接配置：

# application-dev.yml
mqtt:
  enabled: true
  broker: tcp://127.0.0.1:1883
  client-id: fengji-system-dev
  username: test001
  password: test001
  qos: 1
  publish-topic-prefix: fengji/system/
  subscribe-topics:
    - ipc/#          # 订阅所有IPC设备消息
    - fengji/system/# # 订阅系统消息

Topic 设计遵循 层级化、可读性、可扩展性 原则：

Topic 层级结构：
├── ipc/                          # IPC设备消息根节点
│   ├── {deviceId}/               # 设备ID节点
│   │   ├── status/               # 状态类消息
│   │   │   ├── online            # 在线状态
│   │   │   └── running           # 运行状态
│   │   ├── data/                 # 数据类消息
│   │   │   ├── target            # 目标点位数据
│   │   │   ├── conclusion        # 识别结论数据
│   │   │   ├── image             # 图片数据

│   │   ├── camera/               # 摄像头相关

│   │   │   ├── check             # 版本检查

│   │   │   ├── update            # 配置更新

💡 Topic 设计原则：
  • 使用设备ID作为层级：便于按设备订阅和过滤消息
  • 按功能分类：status、data、camera、upgrade 等清晰分类
  • 通配符订阅：使用 ipc/# 订阅所有设备消息
  • 避免特殊字符：Topic 中只使用字母、数字、斜杠、下划线

第三章：MQTT 消息接收与处理

3.1 MqttClientManager 客户端管理

org.jeecg.modules.mqtt.MqttClientManager 负责 MQTT 客户端的生命周期管理，包括连接、断开、重连、消息接收等。

核心流程：

┌─────────────────────────────────────────────────────────────┐
│                    MqttClientManager                          │
│                                                             │
│  1. connect() - 建立MQTT连接                                 │
│     ├── 创建 MqttClient 实例                                │
│     ├── 设置连接选项（用户名、密码、QoS、遗嘱消息等）          │

│                                                             │
│  2. messageArrived() - 消息到达回调                          │
│     ├── 解析 Topic 获取设备ID                               │
│     ├── 记录 MQTT 日志（异步）                               │
│     └── eventPublisher.publishEvent(                        │
│           new MqttMessageEvent(topic, message)              │
│       ) ← Spring事件发布，解耦消息处理                       │
│                                                             │
│  3. connectionLost() - 连接断开回调                          │
│     ├── 记录断开日志                                         │

│                                                             │
│  4. subscribe() - 订阅Topic                                 │
│     ├── 遍历 subscribe-topics 配置                          │

│                                                             │
│  5. 定时重连检查（每30秒）                                   │
│     ├── 检查连接状态                                         │
│     └── 断开则自动重连并重新订阅                             │
└─────────────────────────────────────────────────────────────┘

关键代码片段：

// MqttClientManager.java - 消息到达回调
@Override
public void messageArrived(String topic, MqttMessage message) {
    try {
        // 通过 Spring 事件机制发布消息事件，解耦消息处理逻辑
        eventPublisher.publishEvent(new MqttMessageEvent(this, topic, message));
    } catch (Exception e) {
        logger.error("发布MQTT消息事件失败 - Topic: {}", topic, e);
    }
}
📌 为什么使用 Spring 事件机制？
  • 解耦：MqttClientManager 只负责消息接收，不关心业务处理逻辑
  • 扩展性：可以有多个监听器处理同一事件，互不影响
  • 异步处理：事件监听器可以配置为异步执行，不阻塞消息接收
  • 事务支持：Spring 事件支持事务管理，便于数据一致性控制

3.2 Spring 事件解耦机制

Spring 事件机制是本项目实现 发布-订阅模式 的核心，通过事件将消息接收和业务处理解耦。 

// MqttMessageEvent.java - 自定义事件类
public class MqttMessageEvent extends ApplicationEvent {
    private final String topic;
    private final MqttMessage message;

    public MqttMessageEvent(Object source, String topic, MqttMessage message) {
        super(source);
        this.topic = topic;
        this.message = message;
    }

    public String getTopic() { return topic; }
    public MqttMessage getMessage() { return message; }
}

// MqttMessageHandler.java - 事件监听器
@Component
public class MqttMessageHandler {

    // @EventListener 注解监听 MqttMessageEvent 事件
    @EventListener
    public void onMqttMessageEvent(MqttMessageEvent event) {
        handleMessage(event.getTopic(), event.getMessage());
    }
}

3.4.1 目标点位数据处理链路

这是系统的核心数据流，负责处理设备识别到的净空距离数据。

handleTargetData(deviceId, payload)
    │
    ├── 步骤1: 解析JSON数据
    │   └── JSONObject.parseObject(payload)
    │       ├── distance.physical_distance_m → 净空距离(米)
    │       ├── timestamp → 识别时间戳

    ├── 步骤2: 查询设备信息
    │   └── mqttStatusManager.getDeviceInfo(deviceId)
    │       ├── fanScopeId → 风场ID
    │       ├── fanNo → 风机编号
    │       └── id → 设备ID
    │
    ├── 步骤3: 构建RecordInfo对象
    │   ├── windFarmCode = deviceInfo.getFanScopeId()

    │   ├── recognizeTime = 解析timestamp
    │   ├── shootTime = recognizeTime

    │   └── content = JSON(leaf_tip, boxes, scores)
    │

    │   └── shardingRecordService.saveRecordInfo(recordInfo)
    │       ├── 根据recognizeTime计算日期
    │       ├── 确定分表名: record_info_YYYYMMDD
    │       ├── 确保分表存在(不存在则创建)
    │       └── INSERT INTO record_info_YYYYMMDD (...)
    │
    └── 步骤5: 增量更新每日统计
        └── deviceDailyStatsService.addRecord(recordInfo)
            ├── 查询当日统计记录(按deviceNo + statsDate)
            ├── 不存在则创建新记录

                ├── totalRecords += 1
                ├── successCount += 1

                └── minClearanceValue = Math.min(当前, 新值)

🔍 关键代码解析

// MqttMessageHandler.java - handleTargetData方法
private void handleTargetData(String deviceId, String payload) {
    try {
        // 1. 解析JSON数据
        JSONObject jsonObject = JSONObject.parseObject(payload);
        JSONObject distance = jsonObject.getJSONObject("distance");
        
        // 2. 获取净空距离值
        BigDecimal physicalDistance = distance.getBigDecimal("physical_distance_m");
        
        // 3. 查询设备信息
        FengdianDeviceInfo deviceInfo = mqttStatusManager.getDeviceInfo(deviceId);
        
        // 4. 构建RecordInfo对象
        RecordInfo recordInfo = new RecordInfo();
        recordInfo.setWindFarmCode(deviceInfo.getFanScopeId());
        recordInfo.setWindTurbineCode(deviceInfo.getFanNo());
        recordInfo.setEquipmentCode(deviceInfo.getId());
        recordInfo.setClearanceValue(physicalDistance);
        recordInfo.setRecognizeResult("识别成功");
        
        // 5. 保存到分表
        shardingRecordService.saveRecordInfo(recordInfo);
        
        // 6. 更新每日统计
        deviceDailyStatsService.addRecord(recordInfo);
        
    } catch (Exception e) {
        logger.error("Error handling target data - DeviceId: {}", deviceId, e);
    }
}

    │   └── writeToRedisCache(recordInfo)
    │       ├── Key: record:latest:{windTurbineCode}
    │       ├── Value: JSON(recordInfo)
    │       └── TTL: 60秒
    │
    └── 步骤5: 写Redis ZSet缓存（趋势图）
        └── writeToRedisZSet(recordInfo)
            ├── Key: record:trend:{windTurbineCode}:{yyyyMMdd}
            ├── Score: shootTime毫秒时间戳
            ├── Member: JSON(recordInfo)
            ├── 限制大小: 保留最近1000条
            └── TTL: 2天
📖 Redis缓存设计详解
缓存类型 Key格式 数据结构 用途 TTL
实时数据缓存 record:latest:{设备编码} String 存储设备最新一条检测记录，前端实时查询使用 60秒
趋势数据缓存 record:trend:{设备编码}:{日期} ZSet (有序集合) 存储当日所有检测记录，按时间排序，用于绘制趋势图 2天
设备状态缓存 device:status:{设备ID} String 缓存设备在线状态，避免频繁查询数据库 永不过期
时间同步缓存 device:time:sync:last:{设备ID} String 记录上次同步时间戳，控制30分钟同步间隔 30分钟

3.4.3 设备状态更新链路

handleIpcOnlineStatus(deviceId, payload)
    │
    ├── 步骤1: 使用Redis缓存判断状态是否变更
    │   └── mqttStatusManager.handleDeviceStatusUpdate(deviceId, payload, "online")
    │       ├── 查询Redis: device:status:{deviceId}
    │       ├── 比较新旧状态
    │       └── 返回: needUpdate (true/false)
    │
    ├── 步骤2: 如果状态变更，更新数据库
    │   └── updateDeviceStatusInDatabase(deviceId, "online", payload)
    │       ├── 查询设备信息: fengdian_device_info
    │       ├── 更新运行状态: run_status = '1'
    │       ├── 更新最后在线时间: last_time = NOW()
    │       └── 执行更新: UPDATE fengdian_device_info SET ...
    │
    ├── 步骤3: 检查是否需要同步时间
    │   └── checkAndSyncTime(deviceId)
    │       ├── 查询Redis: device:time:sync:last:{deviceId}
    │       ├── 如果未同步或超过30分钟
    │       └── 发送同步时间命令: mqttService.publishSyncTimeCommand(deviceId)
    │
    └── 步骤4: 推送设备状态变更到前端
        └── pushDeviceStatusChange(deviceInfo, status)
            ├── 构建WebSocket消息: {type, deviceId, deviceNo, status, changeTime}
            └── webSocket.sendMessage(message.toJSONString())
💡 为什么使用Redis缓存判断状态变更？
  • 减少数据库写入：设备每30秒发送一次心跳，如果每次都更新数据库，会造成大量无效写入
  • 提升性能：Redis查询比MySQL快100倍以上
  • 降低锁竞争：减少数据库行锁竞争，提升并发性能
  • 节省IO：只有状态真正变更时才写数据库，减少磁盘IO

4.4 增量统计机制

DeviceDailyStatsService 实现了增量统计机制，每次新记录入库时自动更新当日的统计数据。 

// DeviceDailyStatsServiceImpl.java - 增量统计
public void addRecord(RecordInfo recordInfo) {
    try {
        // 1. 查询当日统计记录
        DeviceDailyStats stats = getByDeviceAndDate(
            recordInfo.getEquipmentCode(), 
            recordInfo.getRecognizeTime()
        );
        
        if (stats == null) {
            // 2. 不存在则创建新记录
            stats = new DeviceDailyStats();
            stats.setDeviceNo(recordInfo.getEquipmentCode());
            stats.setStatsDate(recordInfo.getRecognizeTime());
            stats.setTotalRecords(1);
            stats.setAvgClearanceValue(recordInfo.getClearanceValue());
            stats.setMaxClearanceValue(recordInfo.getClearanceValue());
            stats.setMinClearanceValue(recordInfo.getClearanceValue());
            stats.setSuccessCount(1);
            save(stats);
        } else {
            // 3. 存在则增量更新
            int totalRecords = stats.getTotalRecords() + 1;
            stats.setTotalRecords(totalRecords);
            
            // 在线计算平均值（避免全表聚合）
            BigDecimal avg = stats.getAvgClearanceValue();
            BigDecimal newValue = recordInfo.getClearanceValue();
            BigDecimal newAvg = avg.multiply(new BigDecimal(totalRecords - 1))
                .add(newValue)
                .divide(new BigDecimal(totalRecords), 4, RoundingMode.HALF_UP);
            stats.setAvgClearanceValue(newAvg);
            
            // 更新最大值和最小值
            stats.setMaxClearanceValue(stats.getMaxClearanceValue().max(newValue));
            stats.setMinClearanceValue(stats.getMinClearanceValue().min(newValue));
            
            // 更新成功/失败计数
            if ("识别成功".equals(recordInfo.getRecognizeResult())) {
                stats.setSuccessCount(stats.getSuccessCount() + 1);
            } else {
                stats.setFailCount(stats.getFailCount() + 1);
            }
            
            updateById(stats);
        }
    } catch (Exception e) {
        log.error("增量更新每日统计失败", e);
    }
}
📌 增量统计的优势：
  • 实时性：每条记录入库时立即更新统计，无需等待定时任务
  • 高性能：在线计算平均值，避免全表聚合查询
  • 低开销：只更新一条统计记录，而不是扫描全部分表
  • 可追溯：原始数据保留在分表中，统计数据保留在汇总表中

6.1 分表清理定时任务

系统使用 Quartz 定时任务每天凌晨自动清理过期分表。

// ShardingTableCleanJob.java - 分表清理任务
@Component
public class ShardingTableCleanJob implements Job {

    @Autowired
    private IShardingRecordService shardingRecordService;

    private String parameter; // 保留天数参数

    @Override
    public void execute(JobExecutionContext context) {
        log.info("========== 分表清理任务开始 ==========");
        
        // 1. 解析保留天数（默认30天）
        int retentionDays = 30;
        if (StringUtils.isNotBlank(parameter)) {
            retentionDays = Integer.valueOf(parameter);
        }
        
        try {
            // 2. 计算删除截止日期
            LocalDate beforeDate = LocalDate.now().minusDays(retentionDays);
            
            // 3. 清理RecordInfo分表
            int deleted = shardingRecordService
                .deleteRecordInfoTablesBeforeDate(beforeDate);
            log.info("清理RecordInfo分表完成，删除 {} 个表", deleted);
            
            // 4. 清理MQTT日志分表
            deleted = shardingRecordService
                .deleteMqttLogTablesBeforeDate(beforeDate);
            log.info("清理MQTT日志分表完成，删除 {} 个表", deleted);
            
            // 5. 预创建未来3天的分表
            shardingRecordService.preCreateTables(3);
            
        } catch (Exception e) {
            log.error("分表清理任务执行失败", e);
        }
        
        log.info("========== 分表清理任务结束 ==========");
    }
}

@Component
public class DailyStatsCalcJob implements Job {

    @Autowired

    @Override
    public void execute(JobExecutionContext context) {
        log.info("========== 每日统计计算任务开始 ==========");

            // 计算前一天的统计数据
            Date yesterday = new Date(System.currentTimeMillis() - 24 * 60 * 60 * 1000);

            log.error("每日统计计算任务执行失败", e);
        }
        
        log.info("========== 每日统计计算任务结束 ==========");
    }
}
📌 增量统计 vs 定时统计：
  • 增量统计：实时性高，每条记录入库时立即更新，适用于正常运行的系统
  • 定时统计：用于补算历史数据或修复增量统计的偏差，保证数据准确性
  • 双重保障：增量统计保证实时性，定时统计保证准确性

6.3 数据备份与恢复

系统的数据备份策略主要依赖MySQL的备份机制和分表的优势。

📖 数据备份策略
备份类型 备份方式 频率 说明
全量备份 mysqldump 每天凌晨 备份整个数据库，包含所有分表
增量备份 binlog 实时 记录所有数据变更，支持时间点恢复
分表备份 SELECT INTO OUTFILE 按天 利用分表优势，每天备份当天分表

分表备份的优势

  • 灵活备份：可以单独备份某一天的分表，不影响其他数据
  • 快速恢复：恢复单天数据只需导入对应分表，无需全库恢复
  • 降低风险：分表DROP操作不影响其他分表，误删风险低
  • 节省空间：只备份需要的分表，减少备份文件大小

7.4.1 写入性能优化

7.4.3 最佳实践总结

📖 数据流设计最佳实践
  • 解耦设计：使用Spring事件机制解耦消息接收和业务处理，提升可维护性
  • 多级缓存：Redis String + ZSet + MySQL分表，不同场景选择最优方案
  • 增量统计：避免全表聚合，在线计算平均值，保证实时性和性能
  • 分表策略：按天分表，单表数据量可控，查询快、清理快、维护简单
  • 状态缓存：Redis缓存设备状态，减少无效DB写入，降低锁竞争
  • 异步处理：MQTT日志异步写入，不阻塞消息接收主流程
  • 预创建分表：定时任务预创建未来分表，避免并发创建冲突
  • 降级策略：Redis未命中时自动降级查分表，保证数据可用性
  • 并行加载：前端使用Promise.all并行加载多个接口，减少等待时间
  • 分页查询：列表数据使用分页，避免一次性加载大量数据
⚠️ 注意事项：
  • 分表数量控制：保留天数不宜过长，建议30-90天，避免分表数量过多
  • 索引优化：每个分表3-4个索引即可，过多索引影响写入性能
  • Redis内存：合理设置TTL，避免缓存数据无限增长导致内存溢出
  • 并发控制：高并发场景下注意分表创建的并发控制，使用分布式锁或预创建策略
  • 数据一致性：增量统计和定时统计双重保障，定期校验数据准确性
