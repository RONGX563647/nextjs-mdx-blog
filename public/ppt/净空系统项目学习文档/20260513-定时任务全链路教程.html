第一章：定时任务框架概述

1.1 Quartz 调度框架简介

本项目采用 Quartz 作为定时任务调度框架。Quartz 是一个功能丰富的开源任务调度库，广泛应用于 Java 应用中。其核心概念包括：

  • Job（任务）：定义要执行的业务逻辑，实现 org.quartz.Job 接口
  • JobDetail（任务详情）：封装 Job 的元数据，包括任务名称、组名、关联数据等
  • Trigger（触发器）：定义任务的执行时间和频率，包括 SimpleTrigger 和 CronTrigger
  • Scheduler（调度器）：负责管理 Job 和 Trigger，按触发器配置执行任务
  • Cron Expression（Cron 表达式）：用于定义复杂的调度时间规则

1.4 Cron 表达式详解

Cron 表达式是 Quartz 中用于定义任务执行时间的核心配置，由 6-7 个字段组成：

📖 Cron 表达式格式
秒 分 时 日 月 周 [年]
│  │  │  │  │  │   │
│  │  │  │  │  │   └─ 年（可选）
│  │  │  │  │  └───── 周（DAY_OF_WEEK，1-7 或 SUN-SAT）
│  │  │  │  └──────── 月（MONTH，1-12 或 JAN-DEC）
│  │  │  └─────────── 日（DAY_OF_MONTH，1-31）
│  │  └────────────── 时（HOUR_OF_DAY，0-23）
│  └───────────────── 分（MINUTE，0-59）
└──────────────────── 秒（SECOND，0-59）

特殊字符说明：
*  → 匹配所有值
?  → 不指定值（日和周只能用一个）
-  → 范围（如 1-5 表示 1 到 5）
,  → 列举（如 1,3,5 表示 1、3、5）
/  → 增量（如 0/15 表示从 0 开始每 15 个单位）
L  → 最后（如 L 表示最后一天，5L 表示最后一个周四）
W  → 工作日（如 15W 表示离 15 号最近的工作日）
#  → 第几个星期几（如 6#3 表示第三个周五）

第二章：分表清理任务 (ShardingTableCleanJob)

2.1 任务设计思路

2.1.1 为什么需要分表？

风电净空监控系统每天产生大量识别记录和 MQTT 日志数据。如果所有数据都存储在单表中，随着时间推移，表数据量会急剧增长，导致：

  • 查询性能急剧下降：索引膨胀、全表扫描、锁竞争加剧
  • 数据清理困难：DELETE 操作慢且产生大量碎片，需要定期 OPTIMIZE TABLE
  • 备份恢复时间长：单表数据量越大，备份和恢复耗时越长
  • 维护成本高：大表添加索引、修改表结构需要锁表很长时间

按天分表 是解决上述问题的有效方案：每天的数据独立存储在一张表中，查询时只需扫描对应日期的分表，数据清理时直接 DROP 整张表，效率极高。

📖 分表 vs 单表性能对比（假设每天 10 万条记录）
场景 单表方案 分表方案 性能提升
查询当天数据 扫描 3000 万行（1年） 扫描 10 万行（1天） 300 倍
删除 30 天前数据 DELETE 900 万行 + OPTIMIZE DROP 30 张表 100 倍+
添加索引 锁表数小时 单表只需几分钟 10 倍+
备份单天数据 无法单独备份 直接备份对应分表 灵活

2.1.2 分表清理任务的作用

分表清理任务负责：

  1. 清理过期分表：删除指定天数之前的分表，释放存储空间
  2. 预创建未来分表：提前创建未来几天的分表，确保数据写入时表已存在

2.2 分表策略详解

2.2.1 分表命名规则

数据类型 分表命名格式 示例 说明
识别记录 record_info_YYYYMMDD record_info_20260513 存储设备识别记录
MQTT 日志 mqtt_service_log_YYYYMMDD mqtt_service_log_20260513 存储 MQTT 消息日志
📖 分表命名设计原则
  • 可读性：表名包含日期信息，一目了然
  • 可排序：YYYYMMDD 格式保证按字符串排序即按时间排序
  • 可解析：可通过 SQL 函数从表名提取日期：STR_TO_DATE(RIGHT(table_name, 8), '%Y%m%d')
  • 可匹配：使用 LIKE 'record_info_%' 可匹配所有分表

2.3 核心代码逐行解析

2.3.1 任务类定义

// 文件路径：org.jeecg.modules.quartz.job.ShardingTableCleanJob

package org.jeecg.modules.quartz.job;

import lombok.extern.slf4j.Slf4j;
import org.apache.commons.lang3.StringUtils;
import org.jeecg.modules.fengji.info.service.IShardingRecordService;
import org.quartz.Job;
import org.quartz.JobExecutionContext;
import org.quartz.JobExecutionException;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

import java.time.LocalDate;

/**
 * 分表清理任务
 * 在每天凌晨执行，清理指定天数之前的分表
 * 
 * 执行时机：每天凌晨 2:00（Cron: 0 0 2 * * ?）
 * 参数说明：parameter 字段传入保留天数（1-90天，默认30天）
 * 
 * @author fengji-system
 * @see IShardingRecordService
 */
@Slf4j
@Component
public class ShardingTableCleanJob implements Job {

    // 注入分表服务，负责分表的创建、删除、查询操作
    @Autowired
    private IShardingRecordService shardingRecordService;

    /**
     * 保留天数参数，通过 QuartzJob 的 parameter 字段传入
     * Quartz 调度器会通过反射调用 setParameter() 方法注入参数值
     * 
     * 注意：若参数变量名修改，QuartzJobController 中也需对应修改
     */
    private String parameter;

    // Quartz 通过反射调用此方法注入 parameter 参数
    public void setParameter(String parameter) {
        this.parameter = parameter;
    }
}
📌 关键点：
  • 实现 org.quartz.Job 接口，Quartz 通过反射创建任务实例
  • setParameter() 方法由 Quartz 调度器调用，注入数据库配置的 parameter 参数
  • @Component 注解确保 Spring 能扫描到该类，但实例由 Quartz 创建而非 Spring
  • @Slf4j 是 Lombok 注解，自动生成 log 字段

2.3.2 execute() 方法详解

@Override
public void execute(JobExecutionContext jobExecutionContext) throws JobExecutionException {
    log.info("========== 分表清理任务开始 ==========");
    
    // ==================== 步骤1：解析保留天数参数 ====================
    // 默认保留 30 天，防止参数为空或未配置时使用默认值
    int retentionDays = 30;
    try {
        // 检查 parameter 是否非空
        if(StringUtils.isNotBlank(this.parameter)){
            retentionDays = Integer.valueOf(this.parameter);
        }
        // 参数校验：最少保留 1 天，最多保留 90 天
        // 防止误操作删除过多数据或参数被恶意修改
        if(retentionDays < 1){
            retentionDays = 1;
        }else if(retentionDays > 90){
            retentionDays = 90;
        }
    }catch (Exception e){
        // 参数解析失败（如非数字字符），使用默认值 30 天
        log.warn("参数解析失败，使用默认保留天数: 30天", e);
    }
    
    try {
        // ==================== 步骤2：计算删除截止日期 ====================
        // 例如：今天是 2026-05-13，保留 30 天，则 beforeDate = 2026-04-13
        // 所有日期早于 2026-04-13 的分表将被删除
        LocalDate beforeDate = LocalDate.now().minusDays(retentionDays);
        log.info("清理 {} 天前的分表，截止日期: {}", retentionDays, beforeDate);
        
        // ==================== 步骤3：清理 RecordInfo 分表 ====================
        // 删除所有 record_info_YYYYMMDD 格式的分表（日期 < beforeDate）
        int recordInfoDeleted = shardingRecordService
            .deleteRecordInfoTablesBeforeDate(beforeDate);
        log.info("清理RecordInfo分表完成，删除 {} 个表", recordInfoDeleted);
        
        // ==================== 步骤4：清理 MQTT 日志分表 ====================
        // 删除所有 mqtt_service_log_YYYYMMDD 格式的分表（日期 < beforeDate）
        int mqttLogDeleted = shardingRecordService
            .deleteMqttLogTablesBeforeDate(beforeDate);
        log.info("清理MQTT日志分表完成，删除 {} 个表", mqttLogDeleted);
        
        // ==================== 步骤5：预创建未来 3 天的分表 ====================
        // 提前创建今天及未来 3 天的分表，确保数据写入时表已存在
        // 避免高并发时多个线程同时检测表不存在导致重复创建
        shardingRecordService.preCreateTables(3);
        
        log.info("分表清理任务完成");
    } catch (Exception e) {
        // 任务执行失败，记录错误日志并抛出异常
        // Quartz 会标记任务执行失败，可通过日志排查问题
        log.error("分表清理任务执行失败", e);
        throw new JobExecutionException(e);
    }
    
    log.info("========== 分表清理任务结束 ==========");
}

🔍 代码执行流程说明

  1. 参数解析：从 parameter 字段读取保留天数，若为空或解析失败则使用默认值 30 天。参数范围限制在 1-90 天，防止误操作删除过多数据。这是典型的防御性编程实践。
  2. 计算截止日期：使用 LocalDate.now().minusDays(retentionDays) 计算删除截止日期。例如今天是 2026-05-13，保留 30 天，则 beforeDate = 2026-04-13。所有日期早于此日期的分表将被删除。
  3. 清理 RecordInfo 分表：调用 deleteRecordInfoTablesBeforeDate() 删除所有日期早于 beforeDate 的 record_info_YYYYMMDD 表。返回值为实际删除的表数量。
  4. 清理 MQTT 日志分表：调用 deleteMqttLogTablesBeforeDate() 删除所有日期早于 beforeDate 的 mqtt_service_log_YYYYMMDD 表。返回值为实际删除的表数量。
  5. 预创建未来分表：调用 preCreateTables(3) 提前创建今天及未来 3 天的分表（共 4 天 × 2 种表 = 8 张表），确保数据写入时表已存在，避免并发创建冲突。

2.4.1 TableShardingUtil 核心方法

// 文件路径：org.jeecg.modules.fengji.common.TableShardingUtil

package org.jeecg.modules.fengji.common;

import lombok.extern.slf4j.Slf4j;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.jdbc.core.JdbcTemplate;
import org.springframework.stereotype.Component;

import java.time.LocalDate;
import java.time.format.DateTimeFormatter;
import java.util.List;

/**
 * 分表工具类
 * 提供分表名称生成、存在性检查、自动创建、删除等功能
 * 
 * 设计原则：
 * 1. 所有方法都是线程安全的
 * 2. 使用 JdbcTemplate 直接执行 SQL，不依赖 ORM 框架
 * 3. 异常统一捕获并记录日志，不向上抛出
 */
@Slf4j
@Component
public class TableShardingUtil {

    @Autowired
    private JdbcTemplate jdbcTemplate;

    // 日期格式：yyyyMMdd，如 20260513
    private static final DateTimeFormatter DATE_FORMATTER = 
        DateTimeFormatter.ofPattern("yyyyMMdd");

    // ==================== 分表名称生成方法 ====================
    
    /**
     * 获取指定日期的 RecordInfo 分表名
     * @param date 日期
     * @return 表名，如 record_info_20260513
     */
    public static String getRecordInfoTableName(LocalDate date) {
        return "record_info_" + date.format(DATE_FORMATTER);
    }

    /**
     * 获取指定日期的 MQTT 日志分表名
     * @param date 日期
     * @return 表名，如 mqtt_service_log_20260513
     */
    public static String getMqttLogTableName(LocalDate date) {
        return "mqtt_service_log_" + date.format(DATE_FORMATTER);
    }

    // ==================== 分表存在性检查 ====================
    
    /**
     * 检查表是否存在
     * @param tableName 表名
     * @return true-存在，false-不存在
     */
    public boolean tableExists(String tableName) {
        try {
            // 查询 information_schema.tables 获取表信息
            // DATABASE() 返回当前数据库名
            String sql = "SELECT COUNT(*) FROM information_schema.tables " +
                       "WHERE table_schema = DATABASE() AND table_name = ?";
            Integer count = jdbcTemplate.queryForObject(sql, Integer.class, tableName);
            return count != null && count > 0;
        } catch (Exception e) {
            log.error("检查表格是否存在失败: {}", tableName, e);
            return false;
        }
    }

    // ==================== 分表自动创建 ====================
    
    /**
     * 确保 RecordInfo 分表存在，不存在则自动创建
     * @param date 日期
     */
    public void ensureRecordInfoTableExists(LocalDate date) {
        String tableName = getRecordInfoTableName(date);
        if (!tableExists(tableName)) {
            createRecordInfoTable(tableName);
        }
    }

    /**
     * 创建 RecordInfo 分表
     * 使用 CREATE TABLE IF NOT EXISTS 保证幂等性
     */
    private void createRecordInfoTable(String tableName) {
        try {
            String sql = "CREATE TABLE IF NOT EXISTS `" + tableName + "` (" +
                "`id` varchar(36) NOT NULL COMMENT '主键', " +
                "`wind_farm_code` varchar(100) DEFAULT NULL, " +
                "`wind_turbine_code` varchar(100) DEFAULT NULL, " +
                "`equipment_code` varchar(100) DEFAULT NULL, " +
                "`clearance_value` decimal(10,4) DEFAULT NULL, " +
                "`recognize_time` datetime DEFAULT NULL, " +
                "`shoot_time` datetime DEFAULT NULL, " +
                "`recognize_result` varchar(50) DEFAULT NULL, " +
                "`content` text, " +
                "`original_image` varchar(500) DEFAULT NULL, " +
                "`del_flag` varchar(1) DEFAULT '0', " +
                "`create_time` datetime DEFAULT NULL, " +
                "`update_time` datetime DEFAULT NULL, " +
                "PRIMARY KEY (`id`), " +
                "KEY `idx_equipment_code` (`equipment_code`), " +
                "KEY `idx_recognize_time` (`recognize_time`), " +
                "KEY `idx_wind_turbine_code` (`wind_turbine_code`) " +
                ") ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci";
            jdbcTemplate.execute(sql);
            log.info("创建RecordInfo分表成功: {}", tableName);
        } catch (Exception e) {
            log.error("创建RecordInfo分表失败: {}", tableName, e);
            throw new RuntimeException("创建分表失败: " + tableName, e);
        }
    }

    // ==================== 分表删除 ====================
    
    /**
     * 删除指定日期之前的所有 RecordInfo 分表
     * @param beforeDate 截止日期
     * @return 删除的表数量
     */
    public int deleteRecordInfoTablesBeforeDate(LocalDate beforeDate) {
        List<String> tables = getRecordInfoTablesBeforeDate(beforeDate);
        int deletedCount = 0;
        for (String tableName : tables) {
            if (dropTable(tableName)) {
                deletedCount++;
            }
        }
        log.info("删除RecordInfo分表完成，共删除 {} 个表", deletedCount);
        return deletedCount;
    }

    /**
     * 删除单张分表
     */
    public boolean dropTable(String tableName) {
        try {
            if (tableExists(tableName)) {
                String sql = "DROP TABLE `" + tableName + "`";
                jdbcTemplate.execute(sql);
                log.info("删除分表成功: {}", tableName);
                return true;
            }
            return false;
        } catch (Exception e) {
            log.error("删除分表失败: {}", tableName, e);
            return false;
        }
    }

    /**
     * 预创建未来几天的分表
     * @param days 未来天数
     */
    public void preCreateTables(Integer days) {
        for (int i = 0; i <= days; i++) {
            LocalDate date = LocalDate.now().plusDays(i);
            ensureRecordInfoTableExists(date);
            ensureMqttLogTableExists(date);
        }
    }
}

第三章：每日统计计算任务 (DailyStatsCalcJob)

3.1 任务设计思路

每日统计计算任务在每天凌晨 00:05 执行，完成两个核心功能：

  1. 重算前一天数据：从分表全量查询，按小时分组统计，保证准确性
  2. 补偿缺失数据：检查最近 3 天（不包括今天）的统计记录完整性，按小时级检查并补偿
💡 设计优势：
  • 按小时统计：比按天统计更细粒度，支持小时级数据查询
  • 重算机制：每天重算前一天，保证数据准确性
  • 补偿机制：自动检测并补偿缺失的小时统计数据
  • 双重保障：实时增量统计 + 定时重算，保证数据完整准确

3.2 统计模型设计

统计数据存储在 device_daily_stats 表中，按设备 + 日期 + 小时三维聚合。

3.3 核心代码逐行解析

// DailyStatsCalcJob.java
@Component
public class DailyStatsCalcJob implements Job {

    @Autowired
    private IDeviceDailyStatsService deviceDailyStatsService;

    @Override
    public void execute(JobExecutionContext context) {
        log.info("========== 每日统计计算任务开始 ==========");
        
        try {
            // 计算前一天的统计数据
            Date yesterday = new Date(System.currentTimeMillis() - 24 * 60 * 60 * 1000);
            
            // 遍历所有设备，计算每个设备的统计数据
            List<FengdianDeviceInfo> devices = deviceService.list();
            for (FengdianDeviceInfo device : devices) {
                deviceDailyStatsService.calculateDeviceDailyStats(
                    device.getDeviceNo(), yesterday);
            }
            
            log.info("每日统计计算任务完成");
        } catch (Exception e) {
            log.error("每日统计计算任务执行失败", e);
        }
        
        log.info("========== 每日统计计算任务结束 ==========");
    }
}

3.4 增量更新机制

除了定时任务计算，系统还支持增量更新：每次新记录入库时，自动更新当日的统计数据。

💡 增量更新 vs 定时计算：
  • 增量更新：实时性高，每条记录入库时立即更新
  • 定时计算：用于补算历史数据或修复增量统计的偏差
  • 双重保障：增量统计保证实时性，定时统计保证准确性

3.6 在线平均值算法原理

增量更新时使用在线算法计算平均值，避免全表聚合：

新平均值 = (旧平均值 × 旧记录数 + 新值) ÷ 新记录数

示例：
- 第1条记录: avg = 10.5
- 第2条记录: avg = (10.5 × 1 + 12.3) ÷ 2 = 11.4
- 第3条记录: avg = (11.4 × 2 + 9.8) ÷ 3 = 10.9

优势：
- 无需全表聚合查询
- 只需读取当前统计记录
- 计算复杂度 O(1)
- 性能极高
