第一章：为什么需要分库分表

1.1 业务背景与数据增长

风电净空监控系统是一个典型的 物联网数据采集与分析系统，每天产生大量设备识别记录和 MQTT 通信日志。让我们先看看项目的数据规模：

📖 项目数据规模估算
数据类型 单设备日均产生 设备数量 日均总量 月增长量
识别记录 (RecordInfo) 约 500-2000 条 50-100 台 5-20 万条 150-600 万条
MQTT 日志 (MqttServiceLog) 约 2000-5000 条 50-100 台 20-50 万条 600-1500 万条
每日统计 (ClearanceDailyStats) 1 条/设备 50-100 台 50-100 条 1500-3000 条

按照这个增长速度，一年后的数据量：

  • 识别记录表：约 1800-7200 万条记录
  • MQTT 日志表：约 7200-1.8 亿条记录
  • 总数据量：接近 2 亿条记录，存储约 200-500 GB

1.2 单表瓶颈分析

如果所有数据都存储在单表中，会遇到以下严重问题：

📖 单表性能瓶颈详解

1.2.1 查询性能急剧下降

MySQL InnoDB 引擎使用 B+ 树索引，当表数据量达到千万级时：

  • 索引深度增加：100 万行约 2-3 层，1000 万行约 3-4 层，查询 IO 次数增加
  • 全表扫描代价高：即使有索引，范围查询仍需扫描大量数据页
  • 缓存命中率降低：热数据无法完全加载到 Buffer Pool，频繁磁盘 IO
  • 锁竞争加剧：高并发写入时行锁等待时间增长

1.2.2 数据清理困难

业务要求保留最近 30 天的详细记录，删除历史数据时：

  • DELETE 操作慢：删除 900 万条记录需要数小时，且产生大量 undo log
  • 表碎片严重：DELETE 后表空间不会自动回收，需要 OPTIMIZE TABLE
  • OPTIMIZE 锁表：优化表期间表被锁定，影响业务读写

1.2.3 维护成本高昂

  • 添加索引耗时：千万级表添加索引需要锁表数小时
  • 备份恢复慢：全量备份需要数小时，恢复时间更长
  • 主从延迟：大事务导致主从同步延迟严重
⚠️ 实际案例：某风电场项目单表 record_info 达到 5000 万条记录后，按设备编号查询当天数据从 50ms 飙升到 3-5 秒，严重影响用户体验。

1.3 分库分表 vs 其他方案

面对数据量增长，常见的解决方案对比如下：

方案 适用场景 优点 缺点 本项目选择
SQL 优化 + 索引 数据量 < 1000 万 简单、无需改架构 提升有限，无法解决根本问题 基础方案，必须做
读写分离 读多写少（8:2 以上） 提升读性能，架构简单 不解决单表写入瓶颈和存储问题 配合使用
缓存 (Redis) 热点数据查询 查询极快，减轻数据库压力 不解决存储问题，数据一致性复杂 配合使用
归档表 历史数据很少查询 实现简单，迁移成本低 仍是大表，只是数据量减少 不适合
按天分表（本项目） 数据有明显时间维度 查询快、清理快、维护简单 跨天查询需 UNION，表数量多 ✓ 已采用
ShardingSphere 分库分表 数据量极大，需水平扩展 自动路由、支持分库、透明化 学习成本高、配置复杂、有性能损耗 备选方案
ES/MongoDB 非结构化数据、全文检索 水平扩展能力强 引入新组件、运维成本增加 不适合当前场景
💡 为什么本项目选择按天分表？
  • 业务特性匹配：识别记录和 MQTT 日志都有明确的时间属性，查询多为按天或按时间段
  • 数据生命周期清晰：业务只关心最近 30 天的数据，过期数据直接删除即可
  • 实现简单：无需引入额外中间件，使用 JdbcTemplate 动态建表即可
  • 运维成本低：表结构统一，清理任务自动化，无需人工干预
  • 性能提升明显：单表数据量从千万级降到万级，查询性能提升 10-100 倍

第二章：项目分表方案实现

2.1 按天分表策略设计

本项目采用 按天分表 策略，每天的数据独立存储在一张表中。这是最适合当前业务场景的方案。

📖 为什么选择按天而不是按月？
对比维度 按天分表 按月分表
单表数据量 5-20 万条（识别记录） 150-600 万条
查询性能 极快（扫描万级数据） 较快（扫描百万级数据）
数据清理 DROP 30 张表（秒级） DROP 1 张表（但表很大）
表数量 30-90 张（保留 30-90 天） 3-6 张（保留 3-6 月）
跨天查询 需 UNION 多张表 单表查询即可
适用场景 高频按天查询、快速清理 低频查询、表数量少

2.2 分表命名规范

良好的命名规范是分表管理的基础，本项目遵循以下规范：

数据类型 分表命名格式 示例 说明
识别记录 record_info_YYYYMMDD record_info_20260513 存储设备识别记录
MQTT 日志 mqtt_service_log_YYYYMMDD mqtt_service_log_20260513 存储 MQTT 消息日志
📖 命名设计原则
  • 可读性：表名包含日期信息，一目了然
  • 可排序：YYYYMMDD 格式保证按字符串排序即按时间排序
  • 可解析：可通过 SQL 从表名提取日期：STR_TO_DATE(RIGHT(table_name, 8), '%Y%m%d')
  • 可匹配：使用 LIKE 'record_info_%' 可匹配所有分表

2.3 核心工具类 TableShardingUtil

org.jeecg.modules.fengji.common.TableShardingUtil 是分表管理的核心工具类，提供分表名称生成、存在性检查、自动创建、删除等功能。

2.3.1 分表名称生成

package org.jeecg.modules.fengji.common;

import java.time.LocalDate;
import java.time.format.DateTimeFormatter;

public class TableShardingUtil {

    // 日期格式：yyyyMMdd，如 20260513
    private static final DateTimeFormatter DATE_FORMATTER = 
        DateTimeFormatter.ofPattern("yyyyMMdd");

    /**
     * 获取指定日期的 RecordInfo 分表名
     * @param date 日期
     * @return 表名，如 record_info_20260513
     */
    public static String getRecordInfoTableName(LocalDate date) {
        return "record_info_" + date.format(DATE_FORMATTER);
    }

    /**
     * 获取指定日期的 MQTT 日志分表名
     * @param date 日期
     * @return 表名，如 mqtt_service_log_20260513
     */
    public static String getMqttLogTableName(LocalDate date) {
        return "mqtt_service_log_" + date.format(DATE_FORMATTER);
    }
}

2.3.2 分表存在性检查

/**
 * 检查表是否存在
 * @param tableName 表名
 * @return true-存在，false-不存在
 */
public boolean tableExists(String tableName) {
    try {
        // 查询 information_schema.tables 获取表信息
        // DATABASE() 返回当前数据库名
        String sql = "SELECT COUNT(*) FROM information_schema.tables " +
                   "WHERE table_schema = DATABASE() AND table_name = ?";
        Integer count = jdbcTemplate.queryForObject(sql, Integer.class, tableName);
        return count != null && count > 0;
    } catch (Exception e) {
        log.error("检查表格是否存在失败: {}", tableName, e);
        return false;
    }
}
📌 为什么查询 information_schema？
  • SHOW TABLES LIKE 'xxx' 也可以，但 information_schema 更标准、更灵活
  • 可以结合其他条件过滤，如表引擎、表注释等
  • 支持批量查询，一次获取多个表的信息

2.3.3 分表自动创建

/**
 * 确保 RecordInfo 分表存在，不存在则自动创建
 * 使用 CREATE TABLE IF NOT EXISTS 保证幂等性
 */
public void ensureRecordInfoTableExists(LocalDate date) {
    String tableName = getRecordInfoTableName(date);
    if (!tableExists(tableName)) {
        createRecordInfoTable(tableName);
    }
}

private void createRecordInfoTable(String tableName) {
    try {
        String sql = "CREATE TABLE IF NOT EXISTS `" + tableName + "` (" +
            "`id` varchar(36) NOT NULL COMMENT '主键', " +

            "`wind_turbine_code` varchar(100) DEFAULT NULL COMMENT '风机编号', " +
            "`equipment_code` varchar(100) DEFAULT NULL COMMENT '设备编号', " +
            "`clearance_value` decimal(10,4) DEFAULT NULL COMMENT '净空距离值', " +
            "`recognize_time` datetime DEFAULT NULL COMMENT '识别时间', " +
            "`shoot_time` datetime DEFAULT NULL COMMENT '拍摄时间', " +
            "`recognize_result` varchar(50) DEFAULT NULL COMMENT '识别结果', " +
            "`content` text COMMENT '数据内容(JSON)', " +
            "`original_image` varchar(500) DEFAULT NULL COMMENT '原始图片 URL', " +
            "`processed_image` varchar(500) DEFAULT NULL COMMENT '处理后图片 URL', " +
            "`rotation_speed` decimal(10,2) DEFAULT NULL COMMENT '转速值 (r/min)', " +
            "`unbalance_level` varchar(50) DEFAULT NULL COMMENT '不平衡等级', " +
            "`current_wind_speed` decimal(10,2) DEFAULT NULL COMMENT '当前风速 (m/s)', " +
            "`is_abnormal` int DEFAULT 0 COMMENT '是否异常 (0-正常，1-异常)', " +
            "`source_type` varchar(50) DEFAULT NULL COMMENT '来源类型', " +
            "`status_level` varchar(50) DEFAULT NULL COMMENT '状态等级', " +

            "`update_time` datetime DEFAULT NULL COMMENT '更新时间', " +

            "KEY `idx_equipment_code` (`equipment_code`), " +

        jdbcTemplate.execute(sql);
        log.info("创建RecordInfo分表成功: {}", tableName);

        log.error("创建RecordInfo分表失败: {}", tableName, e);
        throw new RuntimeException("创建分表失败: " + tableName, e);

💡 索引设计说明：
  • equipment_code 索引：按设备查询识别记录时使用，高频查询字段
  • recognize_time 索引：按时间范围查询时使用，支持时间区间过滤
  • wind_turbine_code 索引：按风机查询识别记录时使用
  • 索引数量控制：每个分表 3-4 个索引，避免过多索引影响写入性能
  • 新增字段：processed_image（处理后图片）、rotation_speed（转速）、unbalance_level（不平衡等级）等，用于扩展业务功能

2.3.4 分表删除

/**
 * 删除指定日期之前的所有 RecordInfo 分表
 * @param beforeDate 截止日期
 * @return 删除的表数量
 */
public int deleteRecordInfoTablesBeforeDate(LocalDate beforeDate) {
    int deletedCount = 0;
    try {
        // 查询所有 record_info_YYYYMMDD 格式的表，且日期 < beforeDate
        String sql = "SELECT table_name FROM information_schema.tables " +
                "WHERE table_schema = DATABASE() " +
                "AND table_name LIKE 'record_info_%' " +
                "AND STR_TO_DATE(RIGHT(table_name, 8), '%Y%m%d') < ?";
        List<String> tables = jdbcTemplate.queryForList(sql, String.class, beforeDate);
        
        for (String tableName : tables) {
            if (dropTable(tableName)) {
                deletedCount++;
            }
        }
        log.info("删除RecordInfo分表完成，共删除 {} 个表", deletedCount);
    } catch (Exception e) {
        log.error("删除RecordInfo分表失败", e);
    }
    return deletedCount;
}

/**
 * 删除单张分表
 */
public boolean dropTable(String tableName) {
    try {
        if (tableExists(tableName)) {
            String sql = "DROP TABLE `" + tableName + "`";
            jdbcTemplate.execute(sql);
            log.info("删除分表成功: {}", tableName);
            return true;
        }
        return false;
    } catch (Exception e) {
        log.error("删除分表失败: {}", tableName, e);
        return false;
    }
}

2.5 数据写入与查询

2.5.1 数据写入（自动路由到对应分表）

package org.jeecg.modules.fengji.info.service.impl;

import org.springframework.jdbc.core.JdbcTemplate;
import java.sql.Timestamp;
import java.time.LocalDate;
import java.time.ZoneId;
import java.util.Date;

@Service
public class ShardingRecordServiceImpl implements IShardingRecordService {

    @Autowired
    private JdbcTemplate jdbcTemplate;

    @Autowired
    private TableShardingUtil tableShardingUtil;

    /**
     * 根据日期获取对应的分表名，并确保表存在
     */
    private String getRecordInfoTableName(Date date) {
        LocalDate localDate = date.toInstant()
            .atZone(ZoneId.systemDefault())
            .toLocalDate();
        String tableName = TableShardingUtil.getRecordInfoTableName(localDate);
        
        // 确保表存在，不存在则自动创建
        tableShardingUtil.ensureRecordInfoTableExists(localDate);
        
        return tableName;
    }

    @Override
    public boolean saveRecordInfo(RecordInfo recordInfo) {
        try {
            // 根据 recognize_time 路由到对应日期的分表
            String tableName = getRecordInfoTableName(recordInfo.getRecognizeTime());
            
            String sql = "INSERT INTO `" + tableName + "` (" +
                "id, wind_farm_code, wind_turbine_code, equipment_code, " +
                "clearance_value, recognize_time, shoot_time, recognize_result, " +
                "content, original_image, processed_image, " +
                "rotation_speed, unbalance_level, current_wind_speed, is_abnormal, " +
                "source_type, status_level, del_flag, create_time, update_time" +
                ") VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)";
            
            jdbcTemplate.update(sql,
                recordInfo.getId() != null ? recordInfo.getId() : UUID.randomUUID().toString(),
                recordInfo.getWindFarmCode(),
                recordInfo.getWindTurbineCode(),
                recordInfo.getEquipmentCode(),
                recordInfo.getClearanceValue(),
                recordInfo.getRecognizeTime() != null ? new Timestamp(recordInfo.getRecognizeTime().getTime()) : new Timestamp(System.currentTimeMillis()),
                recordInfo.getShootTime() != null ? new Timestamp(recordInfo.getShootTime().getTime()) : new Timestamp(System.currentTimeMillis()),
                recordInfo.getRecognizeResult(),
                recordInfo.getContent(),
                recordInfo.getOriginalImage(),
                recordInfo.getProcessedImage(),
                recordInfo.getRotationSpeed(),
                recordInfo.getUnbalanceLevel() != null ? String.valueOf(recordInfo.getUnbalanceLevel()) : null,
                recordInfo.getCurrentWindSpeed(),
                recordInfo.getIsAbnormal() != null ? recordInfo.getIsAbnormal() : 0,
                recordInfo.getSourceType(),
                recordInfo.getStatusLevel(),
                recordInfo.getDelFlag() != null ? recordInfo.getDelFlag() : 0,
                new Timestamp(System.currentTimeMillis()),
                new Timestamp(System.currentTimeMillis())
            );
            return true;
        } catch (Exception e) {
            log.error("保存RecordInfo到分表失败", e);
            return false;
        }
    }
}
📌 关键设计：
  • 自动路由：根据 recognize_time 自动路由到对应日期的分表
  • 自动建表：写入时如果分表不存在，自动创建，无需人工干预
  • 动态 SQL：表名通过字符串拼接，MyBatis-Plus 的 @TableName 注解不适用此场景
  • 字段扩展：新增 processed_image、rotation_speed、unbalance_level 等字段，支持更丰富的业务场景
  • 空值处理：对于可能为 null 的字段，在插入时进行空值判断并设置默认值
  • UUID 生成：如果 ID 为空，自动生成 UUID 作为主键

2.5.2 数据查询（单天查询）

@Override
public IPage<RecordInfo> queryRecordInfoByDeviceAndTimePaged(
        String deviceNo, Date startTime, Date endTime, 
        int pageNo, int pageSize) {
    try {
        // 根据时间范围确定需要查询的分表
        LocalDate startDate = startTime.toInstant()
            .atZone(ZoneId.systemDefault())
            .toLocalDate();
        LocalDate endDate = endTime.toInstant()
            .atZone(ZoneId.systemDefault())
            .toLocalDate();
        
        // 单天查询直接查对应分表
        if (startDate.isEqual(endDate)) {
            String tableName = TableShardingUtil
                .getRecordInfoTableName(startDate);
            String sql = "SELECT * FROM `" + tableName + "` " +
                "WHERE equipment_code = ? AND recognize_time BETWEEN ? AND ? " +
                "ORDER BY recognize_time DESC LIMIT ?, ?";
            
            List<RecordInfo> records = jdbcTemplate.query(
                sql, new RecordInfoRowMapper(),
                deviceNo, startTime, endTime,
                (pageNo - 1) * pageSize, pageSize
            );
            
            // 构建分页结果
            IPage<RecordInfo> page = new Page<>(pageNo, pageSize);
            page.setRecords(records);
            return page;
        } else {
            // 跨天查询需要 UNION 多张表
            return queryMultiTablePaged(
                deviceNo, startDate, endDate, pageNo, pageSize);
        }
    } catch (Exception e) {
        log.error("分页查询RecordInfo失败", e);
        return new Page<>();
    }
}

2.6 分表清理定时任务

分表清理任务每天凌晨执行，删除过期分表并预创建未来分表。

// ShardingTableCleanJob.java
@Component
public class ShardingTableCleanJob implements Job {

    @Autowired
    private IShardingRecordService shardingRecordService;

    private String parameter; // 保留天数

    public void setParameter(String parameter) {
        this.parameter = parameter;
    }

    @Override
    public void execute(JobExecutionContext context) {
        log.info("========== 分表清理任务开始 ==========");
        
        int retentionDays = 30; // 默认保留30天
        if (StringUtils.isNotBlank(parameter)) {
            retentionDays = Integer.valueOf(parameter);
        }
        
        try {
            LocalDate beforeDate = LocalDate.now().minusDays(retentionDays);
            
            // 清理RecordInfo分表
            int deleted = shardingRecordService
                .deleteRecordInfoTablesBeforeDate(beforeDate);
            log.info("清理RecordInfo分表: {} 个", deleted);
            
            // 清理MQTT日志分表
            deleted = shardingRecordService
                .deleteMqttLogTablesBeforeDate(beforeDate);
            log.info("清理MQTT日志分表: {} 个", deleted);
            
            // 预创建未来3天的分表
            shardingRecordService.preCreateTables(3);
            
        } catch (Exception e) {
            log.error("分表清理任务执行失败", e);
        }
        
        log.info("========== 分表清理任务结束 ==========");
    }
}
💡 分表清理的优势：
  • DROP TABLE 极快：秒级删除整张表，无需DELETE
  • 空间立即回收：DROP后表空间立即释放
  • 无碎片：不会产生表碎片，无需OPTIMIZE
  • 预创建：提前创建未来分表，避免并发创建冲突

第三章：ShardingSphere 分库分表

3.1 ShardingSphere 简介

ShardingSphere 是一套开源的分布式数据库中间件解决方案，提供分库分表、读写分离、分布式事务等功能。

📖 ShardingSphere 核心组件
组件 定位 特点
ShardingSphere-JDBC 轻量级Java框架 以JAR形式提供服务，无需额外部署
ShardingSphere-Proxy 分布式数据库代理 独立部署，支持多语言客户端
ShardingSphere-Sidecar Kubernetes云原生代理 支持Service Mesh架构

3.2 单库分表配置

在 Spring Boot 项目中配置 ShardingSphere-JDBC 实现单库分表（ShardingSphere 5.x）。

# application-sharding.yml
spring:
  shardingsphere:
    props:
      sql-show: true
    datasource:
      names: ds0
      ds0:
        type: com.alibaba.druid.pool.DruidDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        url: jdbc:mysql://127.0.0.1:3306/fengji_system
        username: root
        password: 123456
    
    rules:
      sharding:
        # 分布式序列算法配置
        key-generators:
          snowflake:
            type: SNOWFLAKE
            props:
              worker-id: 123
        # 分片算法配置
        sharding-algorithms:
          table-classbased:
            type: CLASS_BASED
            props:
              strategy: standard
              algorithmClassName: org.jeecg.modules.test.sharding.algorithm.StandardModTableShardAlgorithm
        tables:
          # 逻辑表名称
          sys_log:
            #配置具体表的数据节点
            actual-data-nodes: ds0.sys_log$->{0..1}
            # 分表策略
            table-strategy:
              standard:
                # 分片算法名称
                sharding-algorithm-name: table-classbased
                # 分片列名称 (对应数据库字段)
                sharding-column: log_type
📘 配置说明：
  • ShardingSphere 5.x：使用 rules.sharding 替代了旧版本的 sharding
  • actual-data-nodes：定义数据节点，sys_log$->{0..1} 表示 sys_log0 和 sys_log1 两张表
  • sharding-column：分片键，这里是 log_type 字段
  • algorithmClassName：自定义分片算法类的完整路径
  • sql-show：开启 SQL 日志，便于调试

3.3 分库分表配置

当单库性能达到瓶颈时，可以扩展为分库分表（双库双表）。

# application-sharding-multi.yml
spring:
  shardingsphere:
    props:
      sql-show: true
    datasource:
      ds0:
        driverClassName: com.mysql.cj.jdbc.Driver
        url: jdbc:mysql://127.0.0.1:3306/fengji_system
        type: com.alibaba.druid.pool.DruidDataSource
        username: root
        password: root
      ds1:
        driverClassName: com.mysql.cj.jdbc.Driver
        url: jdbc:mysql://127.0.0.1:3306/fengji_system2
        type: com.alibaba.druid.pool.DruidDataSource
        username: root
        password: root
      names: ds0,ds1
    rules:
      sharding:
        # 配置绑定表，每一行为一组
        binding-tables:
          - sys_log
        # 分布式序列算法配置
        key-generators:
          snowflake:
            type: SNOWFLAKE
            props:
              worker-id: 123
        # 分片算法配置
        sharding-algorithms:
          table-classbased:
            type: CLASS_BASED
            props:
              strategy: standard
              algorithmClassName: org.jeecg.modules.test.sharding.algorithm.StandardModTableShardAlgorithm
          # 通过 operate_type 取模的方式确定数据落在哪个库
          database-inline:
            type: INLINE
            props:
              algorithm-expression: ds$->{operate_type % 2}
        tables:
          sys_log:
            #配置具体表的数据节点
            actual-data-nodes: ds$->{0..1}.sys_log$->{0..1}
            # 分库策略
            database-strategy:
              standard:
                sharding-column: operate_type
                sharding-algorithm-name: database-inline
            # 分表策略
            table-strategy:
              standard:
                sharding-algorithm-name: table-classbased
                sharding-column: log_type
💡 配置说明：
  • actual-data-nodes：ds$->{0..1}.sys_log$->{0..1} 表示 2 个库×2 张表=4 张物理表
  • database-strategy：根据 operate_type 字段取模分库
  • table-strategy：根据 log_type 字段取模分表
  • binding-tables：绑定表，提高关联查询效率

3.4 自定义分片算法

项目中使用的标准取模分片算法实现：

package org.jeecg.modules.test.sharding.algorithm;

import org.apache.shardingsphere.sharding.api.sharding.standard.PreciseShardingValue;
import org.apache.shardingsphere.sharding.api.sharding.standard.RangeShardingValue;
import org.apache.shardingsphere.sharding.api.sharding.standard.StandardShardingAlgorithm;

import java.util.Collection;
import java.util.Properties;

/**
 * 用于处理使用单一键
 * 根据分片字段的值和 sharding-count 进行取模运算
 * SQL 语句中有>，>=, <=，<，=，IN 和 BETWEEN AND 操作符，都可以应用此分片策略。
 */
public class StandardModTableShardAlgorithm 
    implements StandardShardingAlgorithm<Integer> {
    
    private Properties props = new Properties();

    /**
     * 用于处理=和 IN 的分片
     * @param collection 目标分片的集合 (表名)
     * @param preciseShardingValue 逻辑表相关信息
     * @return 目标表名
     */
    @Override
    public String doSharding(Collection<String> collection, 
                               PreciseShardingValue<Integer> preciseShardingValue) {
        for (String name : collection) {
            Integer value = preciseShardingValue.getValue();
            // 根据值进行取模，得到一个目标值
            if (name.indexOf(value % 2 + "") > -1) {
                return name;
            }
        }
        throw new UnsupportedOperationException();
    }

    /**
     * 用于处理 BETWEEN AND 分片
     * 如果不配置 RangeShardingAlgorithm，SQL 中的 BETWEEN AND 将按照全库路由处理
     */
    @Override
    public Collection<String> doSharding(Collection<String> collection, 
                                RangeShardingValue<Integer> rangeShardingValue) {
        // 范围查询返回所有分片
        return collection;
    }

    @Override
    public void init() {
        // 初始化方法
    }

    @Override
    public String getType() {
        return "STANDARD_MOD";
    }

    @Override
    public Properties getProps() {
        return this.props;
    }

    @Override
    public void setProps(Properties properties) {
        this.props = properties;
    }
}
📌 算法说明：
  • PreciseShardingValue：处理 = 和 IN 查询的精确分片
  • RangeShardingValue：处理 BETWEEN AND 范围查询
  • 取模逻辑：value % 2 决定数据落在哪个分片（0 或 1）
  • 表名匹配：通过 indexOf 匹配表名中的数字后缀

3.5 读写分离配置

ShardingSphere 5.x 使用 replica-query 替代了旧版本的 master-slave。

spring:
  shardingsphere:

      names: ds0,ds1
      ds0:
        type: com.alibaba.druid.pool.DruidDataSource

📘 配置说明：

  • replica-data-source-names：从数据源列表（读操作）
  • load-balancer-name：负载均衡算法，支持 ROUND_ROBIN（轮询）和 RANDOM（随机）
  • 读写分离场景：适用于读多写少（8:2 以上）的场景

第四章：分片键选择与路由

4.1 分片键设计原则

📖 分片键选择原则
  • 高频查询字段：选择最常用作查询条件的字段
  • 数据分布均匀：避免数据倾斜，保证各分片数据量均衡
  • 业务关联性强：相关数据尽量在同一分片，减少跨分片查询
  • 时间维度：对于有时间属性的数据，按时间分片是最佳选择

4.2 路由策略对比

路由策略 适用场景 优点 缺点
精确路由 分片键等值查询 定位精准，性能最优 仅支持等值查询
范围路由 分片键范围查询 支持时间范围查询 可能扫描多个分片
广播路由 无分片键查询 保证查询完整性 性能最差，扫描所有分片

4.3 跨分片查询优化

  • 尽量使用分片键作为查询条件
  • 避免跨分片的JOIN操作
  • 使用汇总表减少跨分片查询
  • 合理设置分页大小，避免全量扫描

第五章：分布式主键方案

5.1 主键生成策略对比

策略 优点 缺点 适用场景
数据库自增 简单、有序 分库后冲突 单库场景
UUID 全局唯一 无序、长度长 不推荐
雪花算法 全局唯一、有序、高性能 依赖机器时钟 分布式场景（推荐）
Redis自增 全局唯一、有序 依赖Redis 已有Redis基础设施

5.2 雪花算法原理

雪花算法生成的64位Long型ID：

┌─────────────────────────────────────────────────────────┐
│  1位符号位  │  41位时间戳  │  10位工作机器ID  │  12位序列号  │
│  (0)        │  (毫秒)      │  (机器标识)       │  (同毫秒内)   │
│             │              │                  │             │
│  固定为0    │  2^41毫秒    │  2^10=1024台     │  2^12=4096   │
│  正数       │  ≈69年       │  可配置          │  /毫秒       │
└─────────────────────────────────────────────────────────┘

时间戳：41位，从自定义纪元开始，可用约69年
机器ID：10位，支持最多1024台机器
序列号：12位，同一毫秒内可生成4096个ID

总容量：4096个ID/毫秒 × 1000毫秒 × 69年 ≈ 282万亿个ID

第六章：性能优化与最佳实践

6.1 查询性能优化

优化项 优化前 优化后 提升
单天查询 全表扫描（千万级） 分表扫描（万级） 100-1000倍
跨天查询 UNION多表（慢） 汇总表查询 50-100倍
实时查询 查分表（100-500ms） Redis缓存（<10ms） 10-50倍

6.2 写入性能优化

  • 批量插入：使用 batch insert 减少网络开销
  • 异步写入：MQTT日志异步写入，不阻塞主流程
  • 预创建分表：避免并发创建冲突
  • 连接池优化：合理配置Druid连接池参数

6.3 常见问题排查

问题 原因 解决方案
分表创建失败 权限不足或表名冲突 检查数据库权限和表名
跨分片查询慢 未使用分片键 添加分片键或使用汇总表
数据倾斜 分片键选择不当 重新选择分片键或调整算法

6.4 最佳实践总结

📖 分库分表最佳实践
  • 合理选择分片键：根据查询模式选择高频查询字段
  • 控制分片数量：单库分表建议不超过100张
  • 使用汇总表：跨分片查询使用汇总表提升性能
  • 多级缓存：Redis缓存热点数据，减少数据库压力
  • 定期清理：定时清理过期分表，释放存储空间
  • 监控告警：监控分片数据量，及时预警数据倾斜
  • 灰度迁移：存量数据迁移使用双写+校验+切换策略
  • 备份策略：利用分表优势，按天备份降低风险
⚠️ 注意事项：
  • 分库分表会增加系统复杂度，评估数据量后再决定是否引入
  • 跨分片JOIN、分页、排序等功能受限，需业务层适配
  • 分布式事务性能开销大，尽量避免跨分片事务
  • 数据迁移风险高，务必做好数据校验和回滚方案
