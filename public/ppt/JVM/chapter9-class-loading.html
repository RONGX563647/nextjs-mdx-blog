第一节：加载阶段 (Loading) - 获取字节流的七种途径

加载阶段核心任务

  • 获取二进制字节流：通过类的全限定名获取.class文件的二进制字节流，可通过多种途径获取（见下方详解）
  • 方法区存储：将字节流所代表的静态存储结构转化为方法区的运行时数据结构（JDK 8后为元空间Metaspace）
  • 生成Class对象：在Java堆中生成一个代表这个类的java.lang.Class对象，作为方法区数据的访问入口

第三节：准备阶段 (Preparation) - 内存分配与零值初始化

准备阶段的核心行为

  • 正式分配内存：为类变量（static修饰的变量）在方法区分配内存
  • 设置零值：将类变量设置为其数据类型的零值（不是代码中设定的值！）
  • 类变量专属：只为类变量分配内存，实例变量将在对象实例化时随对象一起在堆中分配

特殊情况：常量

public static final int VALUE = 123;

如果类字段是常量（static final），准备阶段会直接赋值为代码中的值（123），而不是零值。

因为常量在编译期就已经确定，编译器会将常量值存入ConstantValue属性，准备阶段会识别并赋值。

解析的四种类型

  • 类或接口的解析：将符号引用（全限定名如java.lang.String）解析为直接引用（指向InstanceKlass的指针）
  • 字段解析：查找字段在哪个类中声明，解析为直接引用。先在当前类找，再递归查找父类，最后查找实现的接口
  • 类方法解析：与字段解析类似，但只查找类和父类，不查找接口（接口方法单独解析）。找到后返回方法的直接引用
  • 接口方法解析：接口方法不会在父类中查找，只会在当前接口及其父接口中递归查找

🔧 ClassNotFoundException

原因：类加载器找不到指定的类文件
排查：检查classpath、JAR包依赖、类加载器层次
解决：确保类在正确的类路径下，检查双亲委派是否被破坏

🔧 NoClassDefFoundError

原因：编译时存在但运行时找不到类
排查：检查类是否被正确打包，静态初始化是否抛异常
解决：检查<clinit>中的异常，确保依赖完整

🔧 LinkageError

原因：类的版本不兼容或重复加载
排查：检查JAR包冲突、类加载器隔离
解决：使用Maven/Gradle排除冲突依赖，检查OSGi模块

🔧 ExceptionInInitializerError

原因：静态初始化块或静态变量初始化时抛异常
排查：检查静态代码块中的异常，查看getCause()
解决：修复静态初始化逻辑，避免依赖外部资源

💡 类加载问题排查清单

1. 启动参数检查：-verbose:class 查看类加载日志

2. 类加载器树：打印getClass().getClassLoader().getParent() 链

3. 字节码分析：使用javap -v MyClass.class 查看常量池和符号引用

4. JVM工具：jcmd <pid> VM.classloader_stats 查看类加载统计

5. 热部署问题：检查类的版本兼容性，考虑使用Instrumentation API
