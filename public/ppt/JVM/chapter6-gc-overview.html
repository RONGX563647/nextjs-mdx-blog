第六章：垃圾回收概述 - 深入底层原理

垃圾回收（GC）是JVM自动管理内存的机制。本章深入探讨GC的底层原理、对象判定算法和引用类型实现。

6.1 GC的核心问题

  • 哪些内存需要回收？ 堆、方法区（元空间）、直接内存
  • 什么时候回收？ 内存不足、Young区满、显式调用System.gc()
  • 如何回收？ 标记-清除、标记-复制、标记-整理算法
  • 如何判断对象存活？ 引用计数法、可达性分析法

6.2 引用计数法（Reference Counting）

  • 实现原理：对象头中维护reference_count字段，引用+1，失效-1
  • 底层数据结构：对象头中4字节计数器，支持64位引用计数
  • 优点：实时回收、无STW、实现简单、对象死亡时立即释放
  • 致命缺陷：循环引用导致内存泄漏
  • 线程安全开销：增减计数器需要原子操作（CAS或锁）
  • Python采用：Python的垃圾回收使用引用计数+标记清除处理循环引用

// 引用计数法伪代码
class Object {
  int reference_count = 1; // 对象头中的引用计数器
  void* data;
};

void add_reference(Object* obj) {
  atomic_fetch_add(&obj->reference_count, 1); // 原子操作
}

void release_reference(Object* obj) {
  if (atomic_fetch_sub(&obj->reference_count, 1) == 1) {
    delete obj; // 引用为0，立即释放
  }
}

// 循环引用问题
A.next = B; // A引用B，B.ref_count = 1
B.next = A; // B引用A，A.ref_count = 1
// 外部引用断开后，A和B的ref_count仍为1，无法回收！

6.3 可达性分析法（Reachability Analysis）

  • 核心思想：从GC Roots开始向下搜索，形成引用链（Reference Chain）
  • 数据结构：图论中的有向图，节点=对象，边=引用关系
  • 算法复杂度：O(V+E)，V=对象数，E=引用数
  • 搜索算法：深度优先搜索（DFS）或广度优先搜索（BFS）
  • 现代JVM实现：使用位图（Bitmap）标记对象状态，避免修改对象头
  • 三色标记法基础：为后续并发GC提供理论基础

6.5 四种引用类型 - 底层实现

引用类型强度层级（从强到弱）
🔒 强引用 Strong Reference

Object obj = new Object() | 永远不会被GC回收

↓
🛡️ 软引用 SoftReference

内存不足时才回收 | 适合缓存实现

↓
🍃 弱引用 WeakReference

  • 强引用（Strong Reference）：Object obj = new Object()，只要强引用存在，GC永远不会回收
  • 软引用（SoftReference）：内存不足时（-Xmx限制）才会回收，适合实现缓存
  • 弱引用（WeakReference）：无论内存是否充足，下次GC必定回收
  • 虚引用（PhantomReference）：必须配合ReferenceQueue使用，对象被回收时收到通知
  • 底层实现：通过java.lang.ref.Reference类实现，GC时会特殊处理这些引用类型
  • 引用队列：ReferenceQueue用于接收被回收的引用通知
// 引用类型使用示例
// 1. 软引用 - 适合缓存
SoftReference<byte[]> softRef = new SoftReference<>(new byte[1024*1024]);
byte[] data = softRef.get(); // 可能返回null

// 2. 弱引用 - ThreadLocal、WeakHashMap
WeakReference<Object> weakRef = new WeakReference<>(new Object());
System.gc(); // 下次GC必定回收
assert weakRef.get() == null;

// 3. 虚引用 - 监控对象回收
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef = new PhantomReference<>(obj, queue);
// phantomRef.get() 永远返回null
// 对象被回收时，phantomRef会被加入queue

  • 第一次标记：可达性分析发现对象不可达，判断是否需要执行finalize()
  • finalize()判断：对象覆盖了finalize()且未被调用过，放入F-Queue队列
  • 第二次标记：finalize()中对象重新关联（自救），则移除"即将回收"标记
  • 实际回收：第二次标记后仍不可达的对象，真正释放内存
  • 性能警告：finalize()开销大、不确定性强，Java 9标记为deprecated
  • 替代方案：使用Cleaner（JDK 9+）或try-with-resources管理资源
// finalize()方法示例 - 不推荐使用
public class FinalizeDemo {
  private static FinalizeDemo SAVE_HOOK = null;

  @Override
  protected void finalize() throws Throwable {
    super.finalize();
    // 对象自救 - 重新关联到GC Roots
    SAVE_HOOK = this;
  }

  // Java 9+ 推荐使用 Cleaner
  private static final Cleaner CLEANER = Cleaner.create();
  private final Cleaner.Cleanable cleanable;

  public CleanerDemo() {
    cleanable = CLEANER.register(this, () -> {
      // 清理资源
    });
  }
}

  • 回收机制：Cleaner机制监控，当DirectByteBuffer对象被回收时释放Native内存
  • 底层实现：通过Unsafe.allocateMemory()分配，使用幻象引用跟踪

✅ 关键理解

GC调优黄金法则：
1. 大多数对象朝生夕死 - 优化Young区大小和Survivor比例
2. 减少跨代引用 - 降低记忆集扫描开销
3. 避免大对象 - 防止直接进入老年代触发Full GC
4. 合理设置堆大小 - -Xms=-Xmx避免动态扩展
5. 优先选择G1收集器 - JDK 9+默认，平衡吞吐量和延迟
6. 监控GC日志 - 使用GCViewer、GCEasy等工具分析
