第八章：垃圾收集器 - 深入底层原理

JVM提供了多种垃圾收集器，各有特定优化目标。本章深入剖析每种收集器的底层实现原理。

8.1 垃圾收集器发展历程

JDK 1.0 - Serial收集器

单线程串行执行GC，所有用户线程暂停（STW），简单但停顿时间长

JDK 1.4 - ParNew收集器

Serial的多线程版本，支持并发执行，通常配合CMS使用

JDK 1.4 - Parallel Scavenge收集器

吞吐量优先的多线程新生代收集器，自适应调节新生代大小

JDK 5 - CMS收集器

首次实现并发标记-清除，低延迟但产生碎片，JDK 9废弃

JDK 7 - G1收集器

Region分区+并发标记+可预测停顿，JDK 9成为默认收集器

JDK 11/15 - ZGC收集器

染色指针+读屏障，停顿时间<10ms，支持TB级堆内存

JDK 11 - Shenandoah收集器

Brooks Pointers+读写屏障，Red Hat开发，类似ZGC的设计思路

8.2 Serial收集器 - 最基础的单线程收集器

  • 工作模式：单线程串行执行GC，所有用户线程必须暂停（STW）
  • 新生代算法：标记-复制算法（Serial Copy）
  • 老年代算法：标记-整理算法（Serial Old）
  • 优点：简单高效，无线程切换开销，单核CPU性能最佳
  • 缺点：停顿时间长，不适合交互式应用
  • 适用场景：客户端应用、小型应用、单核环境
  • 启用参数：-XX:+UseSerialGC

// Serial GC伪代码 - 单线程执行
void serialGC() {
  stopTheWorld(); // 暂停所有用户线程

  // 新生代GC - 标记复制
  if (edenFull) {
    youngGC(); // 单线程执行标记-复制
  }

  // 老年代GC - 标记整理
  if (oldGenFull) {
    oldGC(); // 单线程执行标记-整理
  }

  resumeWorld(); // 恢复所有用户线程
}

8.3 ParNew收集器 - Serial的多线程版本

  • 工作模式：多线程并行执行GC，仍需STW暂停用户线程
  • 线程数控制：-XX:ParallelGCThreads=N（默认等于CPU核数）
  • 与Serial区别：仅多线程执行GC，算法逻辑完全相同
  • 配合CMS：JDK 8时代CMS只能配合ParNew使用
  • 工作窃取：空闲线程从其他线程的任务队列窃取任务，提高并行效率
  • 启用参数：-XX:+UseParNewGC（JDK 8及之前）

8.4 Parallel Scavenge收集器 - 吞吐量优先

  • 设计目标：最大化吞吐量（用户代码运行时间占比）
  • 自适应调节策略：自动调整新生代大小、Survivor比例、对象晋升年龄
  • 控制参数：
      -XX:MaxGCPauseMillis=N（最大停顿时间，不保证达到）
      -XX:GCTimeRatio=N（吞吐量目标，默认19即95%）
      -XX:+UseAdaptiveSizePolicy（开启自适应调节，默认开启）
  • 老年代配合：Parallel Old收集器（标记-整理算法）
  • 适用场景：后台计算型应用，对响应时间不敏感
  • 启用参数：-XX:+UseParallelGC

  • 初始标记（Initial Mark）：标记GC Roots直接引用的对象，STW时间极短
  • 并发标记（Concurrent Mark）：与用户线程并发执行，进行可达性分析
  • 重新标记（Remark）：修正并发标记期间用户线程产生的对象变化，STW
  • 并发清除（Concurrent Sweep）：清理未标记对象，与用户线程并发
  • 并发重置（Concurrent Reset）：重置CMS数据结构，为下次GC准备
  • 增量更新（Incremental Update）：记录并发标记期间新增的引用
  • 致命缺点：
      ① 产生内存碎片（标记-清除算法）
      ② 浮动垃圾（并发清除时新产生的垃圾）
      ③ Concurrent Mode Failure（并发失败触发Full GC）
  • 启用参数：-XX:+UseConcMarkSweepGC（JDK 8）
  • 碎片处理：-XX:+UseCMSCompactAtFullCollection（Full GC后压缩）
// CMS GC伪代码
void cmsGC() {
  // 1. 初始标记（STW，很短）
  stopTheWorld();
  markDirectReferences(gcRoots);
  resumeWorld();

  // 2. 并发标记（与用户线程并发）
  while (hasUnmarkedObjects()) {
    markObjects(); // 并发标记
    // 同时用户线程继续执行
  }

  // 3. 重新标记（STW）
  stopTheWorld();
  processIncrementalUpdateLog(); // 处理增量更新日志
  resumeWorld();

  // 4. 并发清除（与用户线程并发）
  while (hasUnmarkedObjects()) {
    sweepObject(); // 并发清除
  }
}

8.6 G1收集器（Garbage-First）- JDK 9+默认收集器

G1采用Region分区+并发标记+可预测停顿设计，是目前最主流的收集器。

  • Region分区：堆内存划分为多个大小相等的Region（1-32MB，2的幂次方）
  • Region类型：Eden、Survivor、Old、Humongous（大对象，≥Region 50%）
  • 并发标记：基于三色标记法，使用SATB（Snapshot At The Beginning）方案
  • 价值回收：优先回收垃圾最多的Region（Garbage-First）
  • 可预测停顿：-XX:MaxGCPauseMillis=N，G1根据此目标选择回收Region数
  • Remembered Set：每个Region维护RSet，记录引用该Region的其他Region
  • 避免全堆扫描：RSet记录跨Region引用，Young GC无需扫描整个老年代

  • 初始标记（Initial Mark）：标记GC Roots，STW，在Young GC时顺便完成
  • 根区域扫描（Root Region Scanning）：扫描Survivor区域引用的老年代Region，并发
  • 并发标记（Concurrent Mark）：全局并发标记，与用户线程并发执行
  • 重新标记（Remark）：处理SATB日志，STW，使用多线程并行
  • 筛选回收（Cleanup）：统计Region存活数据，选择最优回收集合，STW
  • SATB写屏障：记录标记期间被删除的引用，确保不错误回收
  • Mixed GC：同时回收所有新生代+部分高收益老年代Region
  • 启用参数：-XX:+UseG1GC（JDK 8需显式开启，JDK 9+默认）
// G1写屏障（SATB）伪代码
void g1WriteBarrier(Object* obj, int fieldOffset, Object* newValue) {
  Object* oldValue = obj->getField(fieldOffset);

  // SATB：记录被删除的引用（如果指向白色对象）
  if (concurrentMarking && isMarked(obj) && isWhite(oldValue)) {
    satbBuffer.add(oldValue); // 加入SATB日志缓冲区
    if (satbBuffer.isFull()) {
      flushSatbBuffer(); // 缓冲区满，触发处理
    }
  }

  // 执行赋值
  obj->setField(fieldOffset, newValue);
}

// G1 Mixed GC选择回收集合
void chooseCollectionSet() {
  List<Region> collectionSet = new ArrayList<>();
  // 1. 必须包含所有新生代Region
  collectionSet.addAll(youngRegions);

  // 2. 选择收益最高的老年代Region（垃圾最多）
  sort(oldRegions, byGarbageRatio);
  for (region : oldRegions) {
    if (canFinishInTime(region, maxPauseTime)) {
      collectionSet.add(region);
    } else {
      break; // 超出时间目标，停止选择
    }
  }
}

  • 染色指针（Colored Pointers）：利用64位指针的高4位存储状态信息
  • 指针结构：[42位地址][0未用][1位Finalizable][1位Remapped][1位Marked1][1位Marked0]
  • 读屏障：每次读取对象引用时检查染色位，必要时重新映射
  • 多重映射（Multi-Mapping）：同一物理内存映射到多个虚拟地址空间
  • 并发移动对象：通过读屏障+转发指针实现并发对象移动
  • 暂停时间：始终<10ms，与堆大小无关（已测试过16TB堆）
  • 适用场景：超大堆内存（数百GB-数TB），对延迟极度敏感的应用
  • 启用参数：-XX:+UseZGC（JDK 11实验性，JDK 15+生产就绪）

// 读屏障伪代码
Object* zgcReadBarrier(Object* ref) {
  int markBits = getMarkBits(ref); // 获取染色位

  if (isMarked0(ref)) {
    // 对象需要重新映射
    if (!isRemapped(ref)) {
      // 重新映射到新的地址视图
      remap(ref);
    }
    // 返回重新映射后的引用
    return remapReference(ref);
  }

  return ref; // 无需重新映射，直接返回
}

8.8 Shenandoah收集器 - Red Hat的超低停顿方案

  • Brooks Pointers：每个对象前加转发指针，通过间接寻址实现并发移动
  • 读写屏障：读屏障处理转发，写屏障维护SATB日志
  • 与ZGC区别：不需要染色指针，使用对象头中的转发指针
  • 暂停时间：与堆大小无关，通常<10ms
  • 适用场景：类似ZGC，但不支持压缩（JDK 15后支持）
  • 启用参数：-XX:+UseShenandoahGC（JDK 11+）

8.10 实战调优建议

收集器选择决策步骤
步骤1: 评估延迟需求
延迟敏感 → ZGC/Shenandoah
可接受停顿 → G1/Parallel
不敏感 → Serial
步骤2: 评估堆大小
< 2GB → Serial
2GB-32GB → G1
32GB-100GB → G1调优或ZGC
> 100GB → ZGC
步骤3: 评估JDK版本
JDK 8 → G1/Parallel/CMS
JDK 11+ → G1/ZGC
JDK 21+ → 分代ZGC
  • 小型应用（<2G堆）：Serial GC，简单高效
  • 后台计算型（>4G堆）：Parallel GC，最大化吞吐量
  • Web应用（4G-32G堆）：G1 GC，平衡吞吐量和延迟，JDK 9+默认
  • 大型应用（>32G堆）：G1或ZGC，G1可控停顿，ZGC极致低延迟
  • 超大堆（>100G）：ZGC或Shenandoah，停顿时间与堆大小无关
  • G1调优：-XX:MaxGCPauseMillis=200（默认200ms），根据业务调整
  • ZGC调优：-XX:+UseZGC -XX:+ZGenerational（JDK 21+分代ZGC）
  • 监控工具：-Xlog:gc*，配合GCViewer/GCEasy分析

⚠️ 收集器选择决策树

1. 延迟敏感？ 是 → ZGC/Shenandoah > G1 > CMS（已废弃）
2. 吞吐量优先？ 是 → Parallel GC > G1 > Serial
3. 堆内存大小？
    - <2GB → Serial
    - 2GB-32GB → G1（推荐）或Parallel
    - 32GB-100GB → G1调优或ZGC
    - >100GB → ZGC（首选）
4. JDK版本？ JDK 8 → G1/Parallel，JDK 11+ → G1/ZGC，JDK 21+ → 分代ZGC
