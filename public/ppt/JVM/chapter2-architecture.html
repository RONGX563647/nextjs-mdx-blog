深入JVM内部架构，了解类加载器、运行时数据区、执行引擎、本地接口等核心组件的工作原理与协作机制。

双亲委派模型工作流程
sequenceDiagram participant App as 应用程序 participant AppCL as Application ClassLoader participant ExtCL as Extension ClassLoader participant BootCL as Bootstrap ClassLoader participant Cache as 类缓存 App->>AppCL: 加载MyClass AppCL->>Cache: 检查是否已加载 Cache-->>AppCL: 未加载 AppCL->>ExtCL: 委派父加载器 ExtCL->>Cache: 检查是否已加载 Cache-->>ExtCL: 未加载 ExtCL->>BootCL: 委派父加载器 BootCL->>Cache: 检查是否已加载 Cache-->>BootCL: 未加载 BootCL-->>ExtCL: 无法加载（非核心类） ExtCL-->>AppCL: 无法加载（非扩展类） AppCL->>AppCL: 自己加载MyClass AppCL->>Cache: 缓存加载的类 AppCL-->>App: 返回Class对象
  • 步骤1：当前类加载器收到加载请求
  • 步骤2：检查该类是否已经被加载过（缓存机制）
  • 步骤3：如果未加载，将请求委派给父类加载器
  • 步骤4：父类加载器重复步骤2-3，直到Bootstrap ClassLoader
  • 步骤5：如果父类加载器无法加载，子类加载器尝试自己加载

⚠️ 打破双亲委派的场景

某些框架需要打破双亲委派模型来实现特定功能：
1. Tomcat：每个Web应用使用独立的ClassLoader，实现应用隔离
2. 热部署：自定义ClassLoader重新加载修改后的类
3. OSGi：实现模块化的类加载，支持动态安装/卸载模块
4. 加密类加载：自定义ClassLoader在加载时解密.class文件

// 自定义ClassLoader示例 public class CustomClassLoader extends ClassLoader { protected Class<?> findClass(String name) throws ClassNotFoundException { // 1. 从自定义路径读取字节码 byte[] bytes = loadClassBytes(name); // 2. 将字节码转换为Class对象 return defineClass(name, bytes, 0, bytes.length); } }

  • 新生代（Young Generation）：占堆的1/3，分为Eden区和两个Survivor区（S0、S1）
  • 老年代（Old Generation）：占堆的2/3，存储长期存活的对象
  • 大对象直接进入老年代：通过-XX:PretenureSizeThreshold参数设置阈值

  • JDK 7及以前：永久代（PermGen），使用-XX:MaxPermSize设置大小
  • JDK 8及以后：元空间（Metaspace），使用本地内存，默认无上限
  • 为什么改为Metaspace？：避免PermSize难以调优、字符串常量池移到堆中、支持动态类加载场景

解释器 vs JIT编译器

特性解释器JIT编译器
执行方式逐条解释字节码将字节码编译为机器码
启动速度快慢（需要编译时间）
运行性能较慢快（接近本地代码）
内存占用低高（存储编译后代码）
适用场景启动阶段、非热点代码热点代码（执行频繁）
热点代码探测机制
  • 方法调用计数器：记录方法被调用的次数，超过阈值触发JIT编译
  • 回边计数器：记录循环体执行的次数，检测循环热点
  • 阈值设置：Client模式默认1500次，Server模式默认10000次
  • 编译优化：JIT会进行方法内联、逃逸分析、锁优化等

💡 方法内联（Method Inlining）

方法内联是JIT最重要的优化之一，将小方法的代码直接嵌入调用处，消除方法调用开销：

// 优化前 public int add(int a, int b) { return a + b; } public int calculate(int x, int y) { return add(x, y) * 2; } // JIT内联后（等价于） public int calculate(int x, int y) { return (x + y) * 2; // 直接展开add方法体 }

内联条件：方法足够小（默认<35字节）、不是虚方法、调用频率高

2.6 本地方法接口（JNI）

JNI的工作原理
sequenceDiagram participant Java as Java代码 participant JVM as JVM participant JNI as JNI层 participant Native as C/C++代码 participant OS as 操作系统 Java->>JVM: 调用native方法 JVM->>JNI: 查找本地方法 JNI->>JNI: 加载.so/.dll/.dylib JNI->>Native: 调用C/C++函数 Native->>OS: 执行系统调用 OS-->>Native: 返回结果 Native-->>JNI: 返回处理结果 JNI-->>JVM: 转换返回值 JVM-->>Java: 返回Java对象

JNI（Java Native Interface）是Java与C/C++等本地代码交互的桥梁。当Java程序调用native方法时：

  • 步骤1：JVM通过JNI查找对应的本地方法实现
  • 步骤2：加载本地方法库（.so/.dll/.dylib）
  • 步骤3：建立Java方法签名与C/C++函数的映射
  • 步骤4：执行本地代码，处理参数转换和返回值
JNI的典型应用场景
  • 性能敏感操作：图像处理、加密算法、音视频编解码
  • 系统级操作：文件IO、网络编程、内存映射
  • 调用现有C/C++库：OpenCV、FFmpeg、TensorFlow等
  • 硬件交互：串口通信、设备控制

⚠️ JNI使用注意事项

1. 内存泄漏风险：本地代码分配的内存不会自动回收，需要手动释放
2. 线程安全：本地代码需要处理Java线程模型
3. 调试困难：JNI错误可能导致JVM崩溃，难以定位
4. 跨平台性：需要为每个平台编译不同的本地库

✅ 本章小结

JVM架构由类加载器子系统、运行时数据区、执行引擎和本地方法接口四大核心组件组成。理解各组件的职责和协作机制是掌握JVM调优的基础。本章深入分析了类加载器的双亲委派模型、运行时数据区的内存划分、执行引擎的解释器与JIT编译器、垃圾回收系统的工作原理以及JNI的底层实现机制。下一章我们将深入探讨JVM内存模型的具体细节。
